Team Workflows & CI/CD

Scale Claude Code from a single developer to entire teams and automated pipelines. This chapter covers shared configuration, code review patterns, non-interactive mode, and integrations with GitHub Actions and GitLab CI/CD.

Shared Configuration Advanced

Getting Claude Code to behave consistently across a team starts with committing the right files to version control — and keeping the wrong files out.

What to Commit

File / DirectoryPurposeCommit?
CLAUDE.mdProject context: build commands, code style, conventionsYes
.claude/settings.jsonShared permission rules, deny lists, hooksYes
.claude/skills/Reusable skills (SKILL.md files)Yes
.claude/agents/Custom subagent definitionsYes
.claudeignoreFiles Claude should not readYes

What NOT to Commit

File / DirectoryReason
~/.claude/CLAUDE.mdPersonal global instructions — varies per developer.
.claude/settings.local.jsonDeveloper-specific overrides (personal allow rules, local paths).
.env, .env.*Contains API keys and secrets.
~/.claude/settings.jsonUser-level settings — personal preferences.
Tip

Treat CLAUDE.md like code: review it in PRs, prune it regularly, and test changes by observing whether Claude's behavior actually shifts. A bloated CLAUDE.md causes Claude to ignore your actual instructions.

Add these to your .gitignore:

# .gitignore additions for Claude Code
.claude/settings.local.json
.env
.env.*

Writer/Reviewer Pattern Advanced

One of the most effective team patterns is using separate Claude sessions for writing and reviewing. A fresh context window means the reviewer is not biased by the implementation journey — it evaluates the code on its own merits, just like a human reviewer coming in cold.

Session A — Writer
$ claude Implement a rate limiter for our API endpoints. Use the sliding window algorithm. Add it as Express middleware in src/middleware/rateLimiter.ts. Write tests in src/__tests__/rateLimiter.test.ts. ... Claude implements the feature ...
Session B — Reviewer
$ claude Review the rate limiter implementation in @src/middleware/rateLimiter.ts. Look for edge cases, race conditions, and consistency with our existing middleware patterns. Check test coverage. ... Claude reviews with fresh context ...

You can apply this same pattern with tests: have one session write tests from the spec, then another session write the implementation to pass them. Neither session is biased by the other's approach.

Why This Works

A fresh context window improves review quality because Claude will not be biased toward code it just wrote. The reviewer session sees only the final code, not the iterations and false starts that led to it.

Non-Interactive Mode Advanced Docs

The claude -p "prompt" flag runs Claude Code non-interactively — no session, no UI, just output. This is how you integrate Claude into CI pipelines, pre-commit hooks, and automated scripts.

Basic Usage

# One-off query
claude -p "Explain what this project does"

# Structured output for scripts
claude -p "List all API endpoints as JSON" --output-format json

# Streaming JSON for real-time processing
claude -p "Analyze this log file for errors" --output-format stream-json

Restricting Tool Access

Use --allowedTools to limit what Claude can do in unattended mode. This is essential for safety in automated pipelines:

# Only allow reading and editing — no shell commands
claude -p "Fix the TypeScript errors in src/api/" \
  --allowedTools "Read,Edit,Glob,Grep"

# Allow specific bash commands only
claude -p "Run the test suite and report failures" \
  --allowedTools "Read,Bash(npm test *),Bash(npx jest *)"

Fan-Out Pattern

For large migrations or batch operations, distribute work across parallel Claude invocations:

# Generate a file list, then process each in parallel
claude -p "List all Python files needing migration" > files.txt

for file in $(cat files.txt); do
  claude -p "Migrate $file from unittest to pytest. Return OK or FAIL." \
    --allowedTools "Edit,Read,Bash(pytest *)" &
done
wait
Pro Tip

Test your prompt on 2-3 files first, refine based on what goes wrong, then run at scale. Use --verbose during development and disable it in production.

GitHub Actions Integration Advanced Docs

Claude Code GitHub Actions brings AI-powered automation to your pull request workflow. With a simple @claude mention in any PR or issue, Claude can analyze code, create pull requests, implement features, and fix bugs — all while following your project's CLAUDE.md standards.

Quick Setup

The fastest way to set up is through the CLI:

# Inside an active Claude Code session
/install-github-app

This guides you through installing the GitHub app and configuring the required secrets. Alternatively, set it up manually:

  1. Install the Claude GitHub app: github.com/apps/claude
  2. Add ANTHROPIC_API_KEY to your repository secrets.
  3. Copy the workflow file into .github/workflows/.

Full Workflow Example

Complete .github/workflows/claude.yml 
name: Claude Code
on:
  issue_comment:
    types: [created]
  pull_request_review_comment:
    types: [created]
  pull_request:
    types: [opened, synchronize]

jobs:
  # Respond to @claude mentions in PR/issue comments
  claude-mention:
    if: |
      (github.event_name == 'issue_comment' &&
       contains(github.event.comment.body, '@claude')) ||
      (github.event_name == 'pull_request_review_comment' &&
       contains(github.event.comment.body, '@claude'))
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}

  # Auto-review new PRs
  claude-review:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            Review this pull request for code quality,
            correctness, and security. Analyze the diff,
            then post your findings as review comments.
          claude_args: "--max-turns 5 --model sonnet"

@claude Trigger Examples

Once the workflow is installed, use @claude in any PR or issue comment:

# In a PR comment:
@claude implement this feature based on the issue description
@claude fix the TypeError in the user dashboard component
@claude how should I implement auth for this endpoint?

# Claude automatically analyzes context and responds

Key Configuration Parameters

ParameterDescriptionRequired
anthropic_api_keyClaude API key (use repository secrets)Yes
promptInstructions for Claude (plain text or skill name)No
claude_argsCLI arguments passed to Claude CodeNo
trigger_phraseCustom trigger phrase (default: @claude)No
use_bedrockUse AWS Bedrock instead of Claude APINo
use_vertexUse Google Vertex AI instead of Claude APINo
Cost Awareness

Each Claude interaction in CI consumes API tokens. Set --max-turns to prevent excessive iterations and configure workflow-level timeouts. Use GitHub's concurrency controls to limit parallel runs.

GitLab CI/CD Advanced Docs

Claude Code for GitLab CI/CD follows the same principles as the GitHub integration, adapted for GitLab's pipeline model. The integration is maintained by GitLab and is currently in beta.

Quick Setup

  1. Add ANTHROPIC_API_KEY as a masked CI/CD variable in Settings → CI/CD → Variables.
  2. Add a Claude job to .gitlab-ci.yml.
stages:
  - ai

claude:
  stage: ai
  image: node:24-alpine3.21
  rules:
    - if: '$CI_PIPELINE_SOURCE == "web"'
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
  variables:
    GIT_STRATEGY: fetch
  before_script:
    - apk update
    - apk add --no-cache git curl bash
    - curl -fsSL https://claude.ai/install.sh | bash
  script:
    - >
      claude
      -p "${AI_FLOW_INPUT:-'Review this MR and suggest improvements'}"
      --permission-mode acceptEdits
      --allowedTools "Bash Read Edit Write mcp__gitlab"
      --debug

Claude's changes flow through merge requests, so reviewers see every diff and branch protection rules still apply. For enterprise environments, the integration supports AWS Bedrock and Google Vertex AI with OIDC-based authentication — no static keys needed.

Tip

Use @claude in MR or issue comments to trigger Claude (requires a webhook or mention-driven trigger setup). Claude analyzes the context and proposes changes in a new branch or updates the existing MR.

Agent Teams Advanced Docs

Experimental Feature

Agent teams are experimental and disabled by default. Enable them by setting CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 in your settings or environment. Known limitations exist around session resumption, task coordination, and shutdown behavior.

Agent teams let you coordinate multiple Claude Code instances working together. One session acts as the team lead, coordinating work and synthesizing results. Teammates work independently, each in its own context window, and can communicate directly with each other.

How It Differs from Subagents

AspectSubagentsAgent Teams
ContextOwn context; results return to callerOwn context; fully independent
CommunicationReport results back to main agent onlyTeammates message each other directly
CoordinationMain agent manages all workShared task list with self-coordination
Best forFocused tasks where only results matterComplex work requiring collaboration
Token costLower: results summarized backHigher: each teammate is a separate instance

Best Use Cases

  • Research and review — Multiple teammates investigate different aspects of a problem, then share and challenge findings.
  • New modules or features — Teammates each own a separate piece without stepping on each other.
  • Debugging with competing hypotheses — Test different theories in parallel and converge faster.
  • Cross-layer coordination — Frontend, backend, and tests each owned by a different teammate.

Example

# Enable agent teams
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# Start Claude and describe the team
claude
I'm designing a CLI tool for tracking TODO comments.
Create an agent team to explore this from different angles:
- one teammate on UX
- one on technical architecture
- one playing devil's advocate

Cost and Performance Tips

  • Use Sonnet for teammates — It balances capability and cost for coordination tasks. Reserve Opus for the team lead if needed.
  • Keep teams small — Each teammate runs its own context window, so token usage is roughly proportional to team size. Agent teams use approximately 7x more tokens than standard sessions.
  • Keep spawn prompts focused — Teammates load CLAUDE.md, MCP servers, and skills automatically. Everything in the spawn prompt adds to their context from the start.
  • Clean up when done — Active teammates continue consuming tokens even if idle.
When to Use What

Use subagents for quick, focused tasks that report back (like searching docs or running tests). Use agent teams when workers need to share findings, challenge each other, and coordinate on their own. For sequential tasks or same-file edits, a single session is most effective.
