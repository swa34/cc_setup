Containers & Isolation

Run Claude Code inside a development container for full autonomy with hard security boundaries. Containers are the safest way to let the agent loose on real work without babysitting every permission prompt.

Why Run Claude in a Container Beginner Docs

Claude Code is an agentic tool — it reads files, writes code, runs commands, and iterates. Normally you approve each action through the permission system. But for many workflows, that interruption is the bottleneck. You want Claude to just go.

A development container (devcontainer) gives you that. It wraps Claude in an isolated environment with its own filesystem, network firewall, and user context. Because the container can't damage your host machine, you can safely run with --dangerously-skip-permissions and let Claude work uninterrupted.

Key Concept

Think of it this way: permissions ask "can I do this?" before each action. Containers say "do whatever you want, but only within these walls." Containers trade per-action approval for upfront boundary definition.

When to Use a Container Beginner

Not every task needs a container. Here's a decision framework:

The sweet spot

Containers shine when you want to fire and forget — give Claude a well-scoped task and come back to review the result. Common examples:

  • Large refactors — "Migrate all class components to hooks" across 50+ files
  • Feature implementation — "Build the user settings page with tests" from a spec
  • Test generation — "Add unit tests for every exported function in src/utils/"
  • Bug investigation — "Find and fix the memory leak in the WebSocket handler"
  • CI/CD pipelines — Running Claude in headless mode as part of automated workflows
  • Security-sensitive client work — Isolating one client's code from your other projects

Containers vs Sandboxing Intermediate

Claude Code offers two isolation mechanisms. They solve different problems and can be combined.

Sandbox (/sandbox)Devcontainer
What it isolates The Bash tool only — filesystem paths + network domains The entire environment — filesystem, network, processes, users
Enforcement OS-level (Seatbelt on macOS, bubblewrap on Linux) Container-level (Docker/OCI isolation + iptables firewall)
Setup effort One command: /sandbox Requires Docker + devcontainer config
Permission bypass No — still uses Claude's permission system Yes — safe to use --dangerously-skip-permissions
Best for Day-to-day development with reduced prompts Full autonomy, CI/CD, untrusted repos, team onboarding
Can combine? Yes — you can enable sandboxing inside a container for defense in depth
Rule of Thumb

Sandbox = guardrails for your daily driver. Container = a whole separate car. Use sandbox when you're in the driver's seat. Use a container when you want Claude to drive.

Getting Started Beginner

Prerequisites

  • Docker Desktop (or Docker Engine on Linux)
  • VS Code with the Dev Containers extension
  • An Anthropic API key (or Max subscription for Claude Code)

4 steps to launch

  1. Copy the devcontainer config into your project. You need three files in a .devcontainer/ directory: devcontainer.json, Dockerfile, and init-firewall.sh. Grab the reference implementation from github.com/anthropics/claude-code.
  2. Open in VS Code and press Cmd+Shift+P (or Ctrl+Shift+P) → "Dev Containers: Reopen in Container".
  3. Wait for the build — first time takes a few minutes as it installs Node.js, Claude Code, ZSH, and configures the firewall.
  4. Start Claude in the integrated terminal: 
    # Full autonomy mode — safe because you're inside the container
claude --dangerously-skip-permissions
Warning

Even inside a container, --dangerously-skip-permissions is only safe with trusted repositories. A malicious project could still exfiltrate anything accessible inside the container, including Claude Code credentials. Don't open untrusted repos and skip permissions at the same time without reviewing the code first.

Configuration Breakdown Intermediate

The devcontainer setup has three files, each handling a different layer of the environment:

devcontainer.json — Container settings

This is the control plane. It tells VS Code how to build, mount, and configure the container.

{
  "name": "Claude Code Sandbox",
  "build": {
    "dockerfile": "Dockerfile",
    "args": {
      "TZ": "${localEnv:TZ:America/Los_Angeles}",
      "CLAUDE_CODE_VERSION": "latest"
    }
  },
  "runArgs": [
    "--cap-add=NET_ADMIN",
    "--cap-add=NET_RAW"
  ],
  "customizations": {
    "vscode": {
      "extensions": [
        "anthropic.claude-code",
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode",
        "eamodio.gitlens"
      ]
    }
  },
  "remoteUser": "node",
  "mounts": [
    "source=claude-code-bashhistory-${devcontainerId},target=/commandhistory,type=volume",
    "source=claude-code-config-${devcontainerId},target=/home/node/.claude,type=volume"
  ],
  "workspaceMount": "source=${localWorkspaceFolder},target=/workspace,type=bind,consistency=delegated",
  "workspaceFolder": "/workspace",
  "postStartCommand": "sudo /usr/local/bin/init-firewall.sh",
  "waitFor": "postStartCommand"
}

Key points:

  • NET_ADMIN and NET_RAW capabilities are required for the firewall (iptables)
  • Named volumes persist bash history and Claude config between container restarts
  • Your workspace is bind-mounted at /workspace — changes are reflected on the host
  • The firewall script runs on every container start via postStartCommand

Dockerfile — The image

Built on Node.js 20 with development tools pre-installed:

# What's included:
# - Node.js 20 (base image)
# - Claude Code (npm global install)
# - Git, GitHub CLI (gh), fzf, jq
# - ZSH with Powerlevel10k theme
# - git-delta for better diffs
# - iptables/ipset for the firewall
# - nano, vim as editors

Claude Code is installed globally as a non-root node user. You can pin a specific version by changing the CLAUDE_CODE_VERSION build arg.

init-firewall.sh — Network security

This is what makes the container safe for autonomous operation. The firewall implements a default-deny policy:

Allowed Outbound
  • Anthropic API (api.anthropic.com)
  • GitHub (web, API, git — via dynamic IP resolution)
  • npm registry (registry.npmjs.org)
  • VS Code marketplace & updates
  • DNS and SSH
  • Host network (for Docker communication)
Blocked
  • Everything else — default DROP policy
  • Blocked requests get immediate REJECT (no silent timeout)
  • Verified at startup: must fail to reach example.com and succeed reaching api.github.com
How it works

The script uses ipset with iptables for efficient IP matching. GitHub IPs are fetched dynamically from api.github.com/meta and aggregated into CIDR blocks. Other domains are resolved via DNS at startup. This means the firewall adapts to infrastructure changes without manual IP management.

Customizing Your Container Intermediate

Adding allowed domains

If your workflow needs access to additional services (PyPI, Docker Hub, your company API, etc.), add them to the domain list in init-firewall.sh:

# In init-firewall.sh, add domains to the loop:
for domain in \
    "registry.npmjs.org" \
    "api.anthropic.com" \
    "pypi.org" \
    "files.pythonhosted.org" \
    "your-company-api.example.com" \
    # ... existing domains ...
; do

Adding VS Code extensions

Add extension IDs to the customizations.vscode.extensions array in devcontainer.json:

"extensions": [
  "anthropic.claude-code",
  "ms-python.python",
  "ms-python.vscode-pylance",
  "bradlc.vscode-tailwindcss"
]

Adding system packages

Add to the apt-get install line in the Dockerfile:

RUN apt-get update && apt-get install -y --no-install-recommends \
  # ... existing packages ...
  python3 \
  python3-pip \
  postgresql-client \
  && apt-get clean && rm -rf /var/lib/apt/lists/*

Pinning Claude Code version

{
  "build": {
    "args": {
      "CLAUDE_CODE_VERSION": "1.0.42"
    }
  }
}

Headless & CI/CD Mode Advanced

Containers really shine in automated pipelines where there is no human to approve prompts. You can run Claude headlessly with the -p flag (print mode) to pipe a prompt and get the result:

# Run a one-shot task in the container
docker run --rm \
  --cap-add=NET_ADMIN --cap-add=NET_RAW \
  -e ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY" \
  -v "$(pwd):/workspace" \
  your-claude-image \
  claude -p "Review the code in /workspace/src and report any security issues" \
  --dangerously-skip-permissions

For GitHub Actions, you can use the devcontainer as a custom runner environment:

# .github/workflows/claude-review.yml
name: Claude Code Review
on: [pull_request]

jobs:
  review:
    runs-on: ubuntu-latest
    container:
      image: your-org/claude-code-sandbox:latest
      options: --cap-add=NET_ADMIN --cap-add=NET_RAW
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          sudo /usr/local/bin/init-firewall.sh
          claude -p "Review the changes in this PR for bugs and security issues" \
            --dangerously-skip-permissions
Tip

In CI pipelines, always use a pinned Claude Code version instead of latest to avoid unexpected behavior changes between runs.

Security Considerations Intermediate

The container provides strong isolation, but it is not a security silver bullet. Understand the boundaries:

What the container protects

  • Host filesystem — Claude can only access the bind-mounted workspace, not your home directory or system files
  • Network exfiltration — The firewall blocks connections to any domain not explicitly whitelisted
  • System damage — Claude runs as a non-root user (node) inside the container with no access to the host's Docker socket, SSH keys, or system config
  • Other projects — Each container is isolated; client A's code can't be accessed from client B's container

What it does NOT protect

  • The workspace itself — Claude has full read/write to /workspace since that's the whole point. Review its changes before merging.
  • Claude Code credentials — Your API key or session token is inside the container. A malicious repo could theoretically exfiltrate it to an allowed domain (like GitHub).
  • Allowed domains — Traffic to whitelisted domains is not inspected. Data could be sent to api.github.com if that's in the allowlist.
  • Build-time attacks — If the Dockerfile itself is compromised or pulls malicious packages, the container starts in a bad state.
Bottom Line

Containers make autonomous mode dramatically safer, but they don't make it risk-free. Use --dangerously-skip-permissions with trusted repositories only. Always review Claude's output — git diff is your friend.

Best Practices Intermediate

  1. Commit before you launch — Make sure your working tree is clean so you can easily diff Claude's changes against a known baseline.
  2. Scope the task clearly — Autonomous Claude works best with well-defined prompts. "Fix the login bug described in issue #42" beats "improve the codebase."
  3. Use CLAUDE.md — Your container inherits CLAUDE.md from the workspace. Use it to set coding standards, test commands, and guardrails that Claude should follow.
  4. Pin the version — In CI/CD, pin CLAUDE_CODE_VERSION to avoid surprises.
  5. Minimize the domain allowlist — Only add domains you actually need. Every allowed domain is a potential exfiltration path.
  6. Review before merge — Autonomous doesn't mean unreviewed. Treat Claude's commits like any other PR — read the diff, run the tests, check for security issues.
  7. Use named volumes for persistence — The default config persists bash history and Claude settings between container restarts so you don't lose context.

Graduating: Container → Sandbox → Normal Intermediate Sandboxing Docs

Containers are the starting point, not the destination. As you build trust with Claude's behavior in your codebase, you can gradually reduce the overhead. Think of it as training wheels coming off:

Container
Full isolation
→
Sandbox
OS-level guardrails
→
Normal
Permissions only

Stage 1: Container (maximum isolation)

Start here. Run Claude in a devcontainer with --dangerously-skip-permissions. Watch what it does. Review every diff. This is where you learn what Claude does with autonomy — which commands it runs, which files it touches, how it handles ambiguity.

You're ready to move on when: You've run several sessions and Claude's actions are predictable. You trust the types of commands it runs. You're spending more time waiting for Docker than reviewing diffs.

Stage 2: Sandbox (host machine, OS-level boundaries)

Now you're on your actual machine, but with OS-level guardrails. The sandbox restricts which directories Claude can write to and which network domains it can reach — enforced by the kernel, not just Claude's permission system.

Enable it with /sandbox in a Claude session and choose auto-allow mode for the closest experience to container autonomy:

# Inside a Claude Code session:
/sandbox

Configure boundaries in your project's .claude/settings.json:

{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": [".", "/tmp/build"],
      "denyRead": ["~/.ssh", "~/.aws", "~/.gnupg"]
    }
  }
}
What changes from container to sandbox

Gone: Docker overhead, container startup time, separate filesystem image.
Kept: Filesystem write restrictions, network domain filtering, OS-level enforcement on all child processes.
New: You need to explicitly define boundaries (the container did this implicitly via isolation). On Linux, install bubblewrap and socat first.

You're ready to move on when: You've been running sandboxed for a while with no surprises. You know exactly which commands Claude needs to run and which paths it writes to. The sandbox violation prompts have stopped because your config covers the workflow.

Stage 3: Normal (permissions only)

At this point, you trust Claude with your codebase and the permission system is enough. You're in the driver's seat — Claude asks before shell commands and file edits, you approve or deny.

This doesn't mean zero safety. Set up allow/deny rules for your common workflows:

{
  "permissions": {
    "allow": [
      "Bash(npm test *)",
      "Bash(npm run build)",
      "Bash(git diff *)",
      "Bash(git log *)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)",
      "Read(./.env)",
      "Read(./.env.*)"
    ]
  }
}

You can also use acceptEdits mode to auto-approve file changes while still prompting for shell commands — a good middle ground for daily work:

claude --permission-mode acceptEdits
Windows Users

Sandboxing (Stage 2) requires bubblewrap, which needs Linux kernel features. It works on macOS (via Seatbelt) and Linux/WSL2 (via bubblewrap), but native Windows is not supported yet. WSL1 is also not supported. If you're on Windows without WSL2, stay in Stage 1 (container) for autonomous work and jump to Stage 3 (permissions) for interactive sessions.

You can always go back

These stages aren't permanent. Working on an unfamiliar repo? Spin up the container again. Doing a massive refactor? Turn on sandboxing for the session. The trust progression is about your default workflow, not a one-way door.

Troubleshooting Beginner

Firewall fails to initialize

The container needs NET_ADMIN and NET_RAW capabilities. Make sure your runArgs include both:

"runArgs": ["--cap-add=NET_ADMIN", "--cap-add=NET_RAW"]

On Docker Desktop, ensure "Use kernel networking" is enabled if available in your version.

Claude can't reach the API

Check that api.anthropic.com is in the firewall's domain list. Run curl -v https://api.anthropic.com inside the container to debug. If DNS resolution changed, restart the container to re-resolve IPs.

npm install fails

Ensure registry.npmjs.org is whitelisted. Some packages fetch from CDNs — you may need to add domains like cdn.jsdelivr.net to the allowlist.

Git push/pull fails

GitHub IPs are fetched dynamically at container start. If the GitHub API was unreachable during startup, the firewall won't have the right IPs. Restart the container with internet access available.

Container is slow

On macOS with Docker Desktop, bind mounts can be slow for large node_modules. The consistency=delegated flag in workspaceMount helps. For very large projects, consider using a named volume for node_modules.
