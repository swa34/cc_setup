Hooks, Skills & Plugins

Claude Code's extensibility system — from deterministic lifecycle hooks to on-demand skills, shareable plugins, and external tool connections via MCP.

When to Use What Intermediate

Claude Code has four extension mechanisms that serve different purposes. Use this decision tree to pick the right one:

Hooks Intermediate Docs

Hooks are user-defined shell commands that execute at specific points in Claude Code's lifecycle. Unlike CLAUDE.md instructions (which are advisory), hooks are deterministic — they always run when their event fires, and Claude cannot skip them.

Lifecycle events

Each hook fires on a specific event. The most commonly used events:

EventWhen it firesCommon use
PreToolUseBefore a tool call executesBlock dangerous commands, validate inputs
PostToolUseAfter a tool call succeedsAuto-format edited files, log commands
SessionStartWhen a session begins or resumesInject context after compaction
NotificationWhen Claude needs your attentionDesktop notifications
StopWhen Claude finishes respondingVerify all tasks complete
PermissionRequestWhen a permission dialog appearsAuto-approve safe operations
ConfigChangeWhen a config file changesAudit logging
PreCompactBefore context compactionSave important context

Hook configuration

Hooks are configured in settings.json (user-level at ~/.claude/settings.json or project-level at .claude/settings.json). Here is a complete example that auto-formats files with Prettier after every edit:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}
Key Concept

The matcher field is a regex pattern that filters when the hook fires. For PreToolUse/PostToolUse, it matches the tool name (e.g., Bash, Edit|Write, mcp__.*). For SessionStart, it matches the start reason (startup, resume, compact).

Hook input and output

Hooks receive event data as JSON on stdin and communicate results via exit codes:

  • Exit 0 — allow the action to proceed. Stdout content is added to Claude's context for UserPromptSubmit and SessionStart hooks.
  • Exit 2 — block the action. Write a reason to stderr, and Claude receives it as feedback.
  • Any other exit code — action proceeds but stderr is logged for debugging.
Full example: Block edits to protected files

Save this as .claude/hooks/protect-files.sh and make it executable with chmod +x:

#!/bin/bash
# protect-files.sh — Block edits to sensitive files

INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

PROTECTED_PATTERNS=(".env" "package-lock.json" ".git/")

for pattern in "${PROTECTED_PATTERNS[@]}"; do
  if [[ "$FILE_PATH" == *"$pattern"* ]]; then
    echo "Blocked: $FILE_PATH matches protected pattern '$pattern'" >&2
    exit 2
  fi
done

exit 0

Register it in .claude/settings.json:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/protect-files.sh"
          }
        ]
      }
    ]
  }
}

Hook types beyond shell commands

Besides "type": "command", hooks support three other types:

  • "type": "http" — POST event data to a URL. Useful for shared audit services or cloud functions.
  • "type": "prompt" — Single-turn LLM evaluation. Claude (Haiku by default) makes a yes/no decision. Returns {"ok": true} or {"ok": false, "reason": "..."}.
  • "type": "agent" — Multi-turn verification with tool access. Spawns a subagent that can read files and run commands before deciding.
Pro Tip

Claude can write hooks for you. Try: "Write a hook that runs eslint after every file edit" or "Write a hook that blocks writes to the migrations folder." Run /hooks to browse what is configured.

Skills Intermediate Docs

Skills extend what Claude can do by providing instructions in a SKILL.md file. Claude uses skills automatically when relevant, or you invoke one directly with /skill-name. Each skill lives in its own directory under .claude/skills/:

# Skill directory structure
.claude/skills/
├── fix-issue/
│   └── SKILL.md           # Instructions for fixing GitHub issues
├── api-conventions/
│   └── SKILL.md           # REST API design rules
└── deploy/
    ├── SKILL.md            # Deployment workflow
    ├── scripts/
    │   └── deploy.sh       # Helper script
    └── templates/
        └── release-notes.md

SKILL.md format

A SKILL.md has two parts: YAML frontmatter (between --- markers) that tells Claude when to use the skill, and markdown content with the instructions:

---
name: fix-issue
description: Fix a GitHub issue by number
disable-model-invocation: true
allowed-tools: Read, Grep, Glob, Bash(gh *), Edit, Write
---

Fix GitHub issue $ARGUMENTS following our coding standards.

1. Use `gh issue view $ARGUMENTS` to get the issue details
2. Understand the problem described in the issue
3. Search the codebase for relevant files
4. Implement the fix
5. Write and run tests to verify
6. Create a descriptive commit
7. Push and create a PR linking the issue

Key frontmatter fields

name
Display name and slash command. Invoke with /name.
description
Helps Claude decide when to apply the skill automatically.
disable-model-invocation
Set to true to prevent Claude from auto-loading. Use for workflows with side effects (deploy, commit).
allowed-tools
Tools Claude can use without asking permission when this skill is active.
context
Set to fork to run in an isolated subagent context.
agent
Which subagent type to use when context: fork is set (e.g., Explore, Plan).

Bundled skills

Claude Code ships with powerful built-in skills:

SkillPurpose
/batch <instruction>Orchestrate large-scale parallel changes across a codebase. Decomposes work into 5-30 units, each in an isolated worktree.
/claude-apiLoad Claude API reference for your language. Also auto-activates on Anthropic SDK imports.
/debug [description]Troubleshoot your current session by reading the debug log.
/simplify [focus]Review recently changed files for code reuse, quality, and efficiency issues.
Dynamic Context Injection

Skills support !`command` syntax to run shell commands before the skill content is sent to Claude. For example, !`gh pr diff` fetches the PR diff and injects it into the prompt. This is preprocessing — Claude only sees the final result.

Plugins Advanced Docs

Plugins bundle skills, hooks, agents, and MCP servers into a single installable unit. They are the way to share Claude Code extensions across projects and teams.

What plugins contain

# Plugin directory structure
my-plugin/
├── .claude-plugin/
│   └── plugin.json        # Manifest: name, description, version
├── skills/                # Custom skills (SKILL.md files)
│   └── code-review/
│       └── SKILL.md
├── agents/                # Custom agent definitions
├── hooks/
│   └── hooks.json         # Event handlers
├── .mcp.json              # MCP server configurations
├── .lsp.json              # LSP server configurations
└── settings.json          # Default settings when plugin is enabled

When to use plugins vs standalone config

ApproachSkill namesBest for
Standalone (.claude/)/helloPersonal workflows, project-specific, quick experiments
Plugins/plugin-name:helloSharing with team, distributing to community, reusable across projects

Browse and install plugins with /plugin in Claude Code, or install from a marketplace. Test local plugins during development with claude --plugin-dir ./my-plugin.

MCP Servers Advanced Docs

The Model Context Protocol (MCP) connects Claude to external tools and services. MCP servers provide Claude with new tools it can call — databases, APIs, browsers, monitoring services, and more.

How MCP works

An MCP server is a process that exposes tools over a standard protocol. Claude discovers the available tools at startup and can call them during a session, just like its built-in tools (Read, Write, Bash, etc.).

Adding MCP servers

The fastest way to add an MCP server is with the CLI:

Terminal
$ claude mcp add github --transport stdio -- npx -y @modelcontextprotocol/server-github Added MCP server "github" $ claude mcp add postgres --transport stdio -- npx -y @modelcontextprotocol/server-postgres Added MCP server "postgres" $ claude mcp list github stdio @modelcontextprotocol/server-github postgres stdio @modelcontextprotocol/server-postgres

Manual configuration

You can also configure MCP servers directly in .claude/settings.json or ~/.claude/settings.json:

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "your-token-here"
      }
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/allowed/dir"]
    }
  }
}

Common MCP servers

ServerWhat it provides
GitHubIssues, PRs, repos, code search
PostgreSQLDatabase queries, schema inspection
FilesystemScoped file access outside working directory
Brave SearchWeb search results
Puppeteer / PlaywrightBrowser automation and testing
SentryError tracking and monitoring data
MCP Tool Naming

MCP tools follow the naming convention mcp__<server>__<tool>. For example, mcp__github__search_repositories. You can use this pattern in hook matchers to target specific MCP tools.

Rule of Thumb

If it must happen every time, use a hook. If it should happen when relevant, use a skill. If it connects to an external service, use MCP. If you want to share a bundle, make a plugin.
