Token Management

Context is the most important resource to manage in Claude Code. Learn how to track usage, reclaim context, choose the right model, and keep costs predictable.

Why Context Matters Intermediate

Claude's context window holds your entire conversation: every message you send, every file Claude reads, every command output, your CLAUDE.md, loaded skills, and system instructions. It fills up fast. A single debugging session can consume tens of thousands of tokens.

This matters because LLM performance degrades as context fills. When the window is nearly full, Claude may start "forgetting" earlier instructions or making more mistakes. Managing context isn't just about cost — it directly affects the quality of Claude's work.

The Biggest Mistake

The "kitchen sink session": you start with one task, ask something unrelated, then go back to the first task. Context fills with irrelevant information. Use /clear between unrelated tasks.

Tracking Usage Intermediate

Use the /cost command to see your current session's token usage and estimated cost. This is most relevant for API users; subscription users can use /stats for usage patterns.

Terminal — /cost Output
claude> /cost Session Token Usage Total cost: $0.55 Total duration (API): 6m 19.7s Total duration (wall): 6h 33m 10.2s Total code changes: 47 lines added, 12 lines removed

For continuous monitoring, configure your status line to show context usage. The status line appears at the bottom of the terminal and updates in real time.

// ~/.claude/settings.json
{
  "statusLine": {
    "template": "{{model}} | ctx: {{contextPercent}}% | cost: ${{cost}}"
  }
}

Use /context to see a detailed breakdown of what's consuming space in your context window. MCP servers add tool definitions to every request, so a few servers can consume significant context before you start working. Run /mcp to check per-server costs.

/clear vs /compact Intermediate

These are your two primary tools for managing context. Choosing the right one depends on whether you need continuity.

/clear Resets context entirely. Fresh start. Empty (CLAUDE.md reloaded) Switching to an unrelated task
/compact Summarizes conversation, preserving key decisions and code context Compressed summary Running low on context but still working on the same task

Partial Compaction

Use Esc + Esc or /rewind, select a message checkpoint, and choose Summarize from here. This condenses messages from that point forward while keeping earlier context intact — useful when one part of the conversation went off-track.

For quick questions that don't need to stay in context, use /btw. The answer appears in a dismissible overlay and never enters conversation history.

Subagent Delegation Intermediate

Subagents run in their own context window, completely separate from your main conversation. Their work doesn't bloat your context. When done, they return a summary. This isolation is why subagents are one of the most powerful tools for managing context.

When Claude researches a codebase, it reads many files — all of which consume your context. By delegating investigation to a subagent, the detailed file reads stay in the subagent's context while only a concise summary comes back.

Terminal — Subagent Delegation
claude> Use subagents to investigate how our authentication system handles token refresh, and whether we have any existing OAuth utilities I should reuse. Delegating to subagent... [subagent reads 14 files, traces token flow, checks utils/] Subagent findings: Token refresh: SessionManager.refreshToken() in src/auth/session.ts handles refresh via POST /auth/refresh. Tokens expire after 15min, refresh tokens after 7 days. Existing OAuth: src/utils/oauth.ts has a GoogleOAuth class with authorize() and callback() methods. Currently used only for the admin panel. Could be extended for user-facing auth. Main context: only the summary above was added (~200 tokens) Subagent consumed: ~8,400 tokens (stays isolated)

You can also use subagents for verification after implementing something:

# Post-implementation review in a separate context
use a subagent to review this code for edge cases

Model Selection Intermediate

Claude Code supports multiple models with different tradeoffs. Use /model to switch mid-session, or start with claude --model <alias>.

Model Input / Output Cost (per MTok) Speed Capability Best For
Haiku $0.80 / $4 Fastest Good for simple tasks Simple subagent tasks, quick lookups, boilerplate generation
Sonnet $3 / $15 Fast Strong across most tasks Daily coding, debugging, testing, code review. Default for most work.
Opus $15 / $75 Slower Strongest reasoning Complex architecture decisions, multi-step planning, subtle bugs
The opusplan Alias

Use /model opusplan for an automated hybrid: Opus during Plan Mode (for deep reasoning), then Sonnet during execution (for fast coding). Best of both worlds.

Extended context

Opus 4.6 and Sonnet 4.6 support 1 million token context windows. On Max, Team, and Enterprise plans, Opus automatically gets 1M context. For other plans, use the [1m] suffix:

/model opus[1m]
/model sonnet[1m]

Effort levels

Control how deeply Claude reasons with effort levels: low, medium, high, and max (Opus only). Lower effort is faster and cheaper for straightforward tasks.

/effort low     # Quick tasks, less thinking
/effort medium  # Default balance
/effort high    # Complex reasoning
/effort max     # Deepest reasoning (Opus only, current session only)

Automatic Compaction Advanced

Claude Code automatically compacts conversation history when you approach context limits. It clears older tool outputs first, then summarizes the conversation if needed. Your requests and key code snippets are preserved; detailed instructions from early in the conversation may be lost.

How it triggers

Auto-compaction fires when the context window reaches approximately 80-90% capacity. Claude summarizes what matters most: code patterns, file states, key decisions, and recent changes. Older tool outputs and verbose command results are cleared first.

Customizing compaction in CLAUDE.md

Add a "Compact Instructions" section to your CLAUDE.md to control what survives compaction:

# Compact instructions

When compacting, always preserve:
- The full list of modified files and their paths
- Any test commands and their results
- Architecture decisions made during this session
- The current implementation plan if one exists

CLAUDE.md survives compaction

CLAUDE.md fully survives compaction. After /compact or auto-compaction, Claude re-reads your CLAUDE.md from disk and re-injects it fresh. If an instruction disappeared after compaction, it was given only in conversation, not written to CLAUDE.md. This is why persistent rules belong in CLAUDE.md, not in chat messages.

Hooks for Compaction

You can use hooks to run scripts at compaction time. For example, a PreCompact hook could save a snapshot of the current conversation state, and a PostCompact hook could verify that critical context was preserved. Configure hooks in .claude/settings.json or ask Claude to write them for you.

Token-saving strategies

  • Disable unused MCP servers — each adds tool definitions to every request even when idle. Run /mcp to check costs.
  • Use CLI tools instead of MCP when possible — gh, aws, gcloud are more context-efficient than MCP servers.
  • Install code intelligence plugins for typed languages — a single "go to definition" replaces grepping and reading multiple files.
  • Use preprocessing hooks to filter verbose output before it enters context (e.g., grep test output for failures only).
  • Write specific prompts — "improve this codebase" triggers broad scanning; "add input validation to login in auth.ts" is targeted.

Practical Costs Intermediate

Here's what real-world Claude Code usage costs for API users. Subscription users (Max, Pro, Team) have usage included in their plan.

Cost Benchmarks (API Users)

Average daily cost: $6 per developer per day

90th percentile: Under $12 per developer per day

Monthly estimate: ~$100-200 per developer with Sonnet 4.6

Costs vary with codebase size, query complexity, conversation length, and whether you run automation or multiple parallel sessions.

What drives cost

Factor Impact How to Reduce
Long conversations without clearing High — every message re-sends the full context Use /clear between tasks, /compact during long tasks
Using Opus for everything High — 5x more expensive than Sonnet Use Sonnet for daily work, Opus for complex planning only
Extended thinking on simple tasks Medium — thinking tokens billed as output Use /effort low for straightforward tasks
Many MCP servers connected Medium — tool definitions consume context Disable unused servers; prefer CLI tools
Vague prompts ("improve this codebase") Medium — triggers broad file scanning Be specific about files and scope
Agent teams / parallel sessions High — each session has its own context Keep teams small, use Sonnet for teammates

Background token usage

Claude Code uses a small amount of tokens for background tasks even when idle: conversation summarization for --resume, and command processing. This is typically under $0.04 per session.

Prompt caching

Claude Code automatically uses prompt caching to reduce costs for repeated content like system prompts and CLAUDE.md. Cached tokens cost significantly less than fresh tokens. The cache lasts about 5 minutes, which is why frequent messages in an active session cost less per-message than sporadic ones.

Cost Optimization Checklist

1. Use /clear between unrelated tasks.
2. Default to Sonnet; switch to Opus only for complex reasoning.
3. Use /effort low for simple tasks.
4. Delegate verbose investigations to subagents.
5. Keep CLAUDE.md lean (under 500 lines).
6. Disable unused MCP servers.
7. Write specific prompts that scope the work.
