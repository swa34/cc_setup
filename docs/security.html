Security & Permissions

Claude Code operates with real access to your filesystem, shell, and network. Understanding its permission model is the first step to using it safely — whether you are a solo developer or managing a fleet of engineers.

Permission System Overview Intermediate Docs

Claude Code uses a tiered permission system that balances power and safety. Every tool is classified into a tier, and each tier has a different approval requirement. The design principle is fail-closed: if a tool is not explicitly allowed, Claude will ask before using it.

Tool TypeExamplesApproval Required?"Don't ask again" Behavior
Read-only File reads, Grep, Glob No — always allowed N/A
File modifications Edit, Write, NotebookEdit Yes Persists until session end
Bash commands Shell execution Yes Permanently per project directory and command
MCP tools External server tools Yes Follows server-level rules

Permission rules are evaluated in a strict order: deny → ask → allow. The first matching rule wins, which means deny rules always take precedence — no other setting level can override a deny.

Key Concept

Use /permissions inside Claude Code to view and manage all active permission rules. This UI lists every rule and shows which settings.json file it originates from.

Rules follow the format Tool or Tool(specifier). For example:

  • Bash(npm run build) — matches the exact command npm run build
  • Bash(npm run *) — matches any command starting with npm run
  • Read(./.env) — matches reading the .env file in the current directory
  • mcp__puppeteer__puppeteer_navigate — matches one specific MCP tool
Warning

Read and Edit deny rules apply to Claude's built-in file tools, not to Bash subprocesses. A Read(./.env) deny rule blocks the Read tool but does not prevent cat .env in Bash. For OS-level enforcement that blocks all processes, enable the sandbox.

Permission Modes Intermediate

Claude Code supports several permission modes that determine how much autonomy the agent has. Choosing the right mode depends on your environment, the sensitivity of the codebase, and your comfort level.

The full set of modes you can configure via defaultMode in your settings file:

ModeDescription
defaultStandard behavior: prompts for permission on first use of each tool.
acceptEditsAutomatically approves file edits for the session. Still asks before Bash commands.
planPlan Mode: Claude can analyze but not modify files or execute commands.
dontAskAuto-denies tools unless pre-approved via /permissions or permissions.allow rules.
bypassPermissionsSkips permission prompts except for writes to protected directories (.git, .claude, .vscode, .idea). Only for isolated environments.
Warning

bypassPermissions mode skips most permission prompts. Only use this in containers or VMs where Claude cannot cause real damage. Administrators can disable this mode entirely by setting disableBypassPermissionsMode to "disable" in managed settings.

API Key Management Intermediate

Your API key is the credential that authenticates Claude Code against Anthropic's API (or your cloud provider). Mishandling it can expose your account to unauthorized usage and unexpected costs.

Golden Rules

  • Never hardcode API keys in source code, CLAUDE.md, or shell scripts checked into version control.
  • Use environment variables for local development.
  • Use a secrets manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault) in CI/CD and production.
  • Rotate keys regularly and revoke any key that may have been exposed.

For local development, store your key in a .env file that is excluded from version control:

# .env — NEVER commit this file
ANTHROPIC_API_KEY=sk-ant-api03-...your-key-here...

# For AWS Bedrock (alternative provider)
# AWS_PROFILE=claude-code
# AWS_REGION=us-west-2

# For Google Vertex AI (alternative provider)
# CLOUD_ML_REGION=us-east5
# ANTHROPIC_VERTEX_PROJECT_ID=my-project-123

Make sure .env is in your .gitignore:

# .gitignore
.env
.env.*
*.pem
credentials.json
Tip

For GitHub Actions, always use ${{ secrets.ANTHROPIC_API_KEY }} rather than pasting your key into the workflow YAML. For GitLab, use masked CI/CD variables.

Sandboxing Intermediate Docs

Sandboxing provides OS-level isolation that restricts the Bash tool's filesystem and network access. While permissions control which tools Claude can use, sandboxing enforces boundaries that all child processes inherit — including tools like kubectl, terraform, and npm.

Filesystem Isolation
Read and write access restricted to the current working directory and its subdirectories. Cannot modify files outside the project without explicit permission.
Network Isolation
Only approved domains can be accessed. New domain requests trigger permission prompts. All scripts and subprocesses inherit the same restrictions.
OS-Level Enforcement
macOS uses Seatbelt; Linux/WSL2 uses bubblewrap. Child processes cannot escape the sandbox boundaries.

Enable sandboxing with the /sandbox command, which opens a menu to choose between:

  • Auto-allow mode — Sandboxed commands run automatically without requiring permission. Commands needing access to non-allowed hosts fall back to the regular permission flow.
  • Regular permissions mode — All commands go through the standard permission flow, even when sandboxed.

On Linux/WSL2, install the required packages first:

# Ubuntu/Debian
sudo apt-get install bubblewrap socat

# Fedora
sudo dnf install bubblewrap socat

You can customize sandbox behavior through settings. For example, granting write access to paths outside the project directory:

{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube", "/tmp/build"]
    }
  }
}
Defense in Depth

Permissions and sandboxing are complementary security layers. Permission deny rules block Claude from even attempting to access restricted resources. Sandbox restrictions prevent Bash commands from reaching resources outside defined boundaries, even if a prompt injection bypasses Claude's decision-making. Use both for maximum protection.

Protecting Sensitive Files Intermediate

Claude Code automatically reads files it needs for context. Without proper guards, this can expose secrets, credentials, and other sensitive data that should never enter an AI conversation.

The .env Auto-Loading Concern

Claude Code does not automatically execute .env files, but it can read them if it determines they are relevant context. If Claude reads a .env containing API keys, those keys become part of the conversation context and are sent to the API.

Deny Rules in settings.json

The most direct defense is adding deny rules to your project's .claude/settings.json:

{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./**/*.pem)",
      "Read(./**/credentials.json)",
      "Edit(./.env)",
      "Edit(./.env.*)",
      "Edit(./secrets/**)"
    ]
  }
}

These rules block Claude's built-in Read and Edit tools from accessing the matched paths. Remember that Read/Edit deny rules do not block Bash subprocesses — for OS-level enforcement, enable the sandbox with denyRead paths.

.claudeignore

Create a .claudeignore file in your project root, following the same syntax as .gitignore. Files matched by .claudeignore are excluded from Claude's automatic context gathering:

# .claudeignore
.env
.env.*
secrets/
*.pem
**/credentials.json
**/service-account-key.json
node_modules/

.gitignore Synergy

If a file is already in your .gitignore, Claude Code is less likely to surface it. But .gitignore alone is not a security boundary — a .claudeignore or deny rule provides explicit protection.

Warning

Never rely on a single layer. Combine deny rules, .claudeignore, sandbox denyRead paths, and .gitignore for defense in depth. If a secret is accidentally committed to git history, rotating the secret is the only safe remedy.

Team Security Advanced Docs

For organizations with multiple developers using Claude Code, centralized security configuration ensures consistent policy enforcement across the team.

Managed Settings

Administrators can deploy managed settings that cannot be overridden by user or project settings. These follow the same format as regular settings files and can be delivered through:

  • Server-managed settings — Configured via the admin console at Claude.ai, delivered automatically when users authenticate. Best for organizations without MDM infrastructure.
  • Endpoint-managed settings — Deployed via MDM/OS-level policies (macOS managed preferences, Windows registry) or managed settings files. Stronger enforcement on managed devices.

Managed-Only Settings

Some settings are only effective in managed settings, giving administrators exclusive control:

SettingEffect
disableBypassPermissionsModePrevents bypassPermissions mode and the --dangerously-skip-permissions flag.
allowManagedPermissionRulesOnlyOnly managed settings can define allow/ask/deny rules.
allowManagedHooksOnlyOnly managed hooks and SDK hooks are allowed.
allowManagedMcpServersOnlyOnly managed MCP servers are respected.

Shared Permission Configs

Commit a .claude/settings.json to your repository with team-wide deny rules and allowlists. This file applies to all developers who clone the project:

{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(git commit *)",
      "Bash(git diff *)",
      "Bash(npx jest *)"
    ],
    "deny": [
      "Bash(git push *)",
      "Bash(rm -rf *)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}

OpenTelemetry Monitoring

Track Claude Code usage, costs, and tool activity across your organization by exporting telemetry data through OpenTelemetry. Configure via managed settings to ensure all developers emit metrics to your collector:

{
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_METRICS_EXPORTER": "otlp",
    "OTEL_LOGS_EXPORTER": "otlp",
    "OTEL_EXPORTER_OTLP_PROTOCOL": "grpc",
    "OTEL_EXPORTER_OTLP_ENDPOINT": "http://collector.example.com:4317"
  }
}

Available metrics include claude_code.cost.usage, claude_code.token.usage, claude_code.session.count, claude_code.lines_of_code.count, and tool decision events — all segmented by user, session, and model.

ConfigChange Audit Hooks

Use ConfigChange hooks to detect runtime configuration changes. This lets you log modifications or block unauthorized changes before they take effect:

{
  "hooks": {
    "ConfigChange": [
      {
        "matcher": "*",
        "hooks": [
          {
            "type": "command",
            "command": "/usr/local/bin/audit-config-change.sh"
          }
        ]
      }
    ]
  }
}
Deployment Strategy

Start with a small pilot group (5-10 developers) to validate your managed settings and permission rules before rolling out to the full team. Monitor telemetry data during the pilot to identify rules that are too restrictive or too permissive, then adjust before broader deployment.

Settings Precedence

Understanding the hierarchy is critical for team security. Settings are resolved in this order, from highest to lowest priority:

  1. Managed settings — Cannot be overridden by any other level, including CLI arguments.
  2. Command line arguments — Temporary session overrides.
  3. Local project settings (.claude/settings.local.json) — Developer-specific, not committed.
  4. Shared project settings (.claude/settings.json) — Committed to version control.
  5. User settings (~/.claude/settings.json) — Personal global defaults.

If a tool is denied at any level, no other level can allow it. A managed deny always wins.
