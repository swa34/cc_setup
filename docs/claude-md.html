CLAUDE.md Best Practices

CLAUDE.md is the single most important file for shaping Claude Code's behavior. Get it right and Claude follows your conventions automatically. Get it wrong and Claude ignores your rules or wastes context.

What is CLAUDE.md Beginner

New to the .claude/ folder structure? See the Anatomy of .claude/ for a visual overview of every file and where CLAUDE.md fits in.

CLAUDE.md is a markdown file that Claude reads at the start of every conversation. It gives Claude persistent context it cannot infer from code alone: build commands, code style rules, architecture decisions, naming conventions, and workflow instructions.

Think of it as a README for your AI pair programmer. Everything in CLAUDE.md becomes part of Claude's working context, so it shapes every decision Claude makes during the session.

How It Works

CLAUDE.md content is delivered as a user message after the system prompt. Claude reads it and tries to follow it, but it's guidance, not enforced configuration. The more specific and concise your instructions, the more consistently Claude follows them.

Claude also has a complementary system called auto memory, where Claude saves notes for itself as it works (build commands, patterns it discovers, your preferences). Auto memory is stored in ~/.claude/projects/<project>/memory/ and the first 200 lines of MEMORY.md are loaded each session. CLAUDE.md is for your instructions; auto memory is for Claude's learnings.

File Hierarchy Beginner

CLAUDE.md files can live at multiple levels, each with a different scope. More specific locations take precedence over broader ones.

Location Scope Use Case Shared With
~/.claude/CLAUDE.md Global (all projects) Personal preferences, tool shortcuts, code style you always want Just you
./CLAUDE.md or ./.claude/CLAUDE.md Project root Build commands, architecture, team conventions. Check this into git. Team via source control
./subdir/CLAUDE.md Subdirectory Rules specific to a package in a monorepo or a subsystem Team via source control

How files load

Claude Code walks up the directory tree from your working directory and loads every CLAUDE.md it finds. If you run Claude in foo/bar/, it loads both foo/bar/CLAUDE.md and foo/CLAUDE.md.

CLAUDE.md files in subdirectories below your working directory load on demand when Claude reads files in those directories, not at startup.

Monorepo patterns

In a monorepo, put shared conventions at the root and package-specific rules in subdirectories:

monorepo/
  CLAUDE.md              # Shared: commit style, CI commands, PR conventions
  packages/
    api/
      CLAUDE.md          # API-specific: REST conventions, Prisma commands
    web/
      CLAUDE.md          # Frontend-specific: React patterns, CSS conventions
    shared/
      CLAUDE.md          # Shared library rules

If other teams' CLAUDE.md files get picked up and cause conflicts, use claudeMdExcludes in your local settings to skip them:

// .claude/settings.local.json
{
  "claudeMdExcludes": [
    "**/monorepo/other-team/.claude/rules/**"
  ]
}

There is also a managed policy location for organization-wide instructions deployed by IT/DevOps (on Linux: /etc/claude-code/CLAUDE.md). These cannot be excluded by individual settings.

What to Include Intermediate

Include things Claude cannot figure out by reading your code. Focus on commands, conventions, and decisions that are non-obvious.

Build & Test Commands
npm run build, pytest -x, specific flags or env vars needed. Claude will run these to verify its work.
Code Style Rules
Rules that differ from defaults: "Use 2-space indentation", "ES modules only, no CommonJS", "Destructure imports".
Architecture Notes
Handler -> Service -> Model layering, where new files go, naming patterns for routes.
Testing Instructions
Preferred test runner, how to run a single test, co-located vs separate test files.
Common Gotchas
Environment variables that must be set, services that need to be running, non-obvious behaviors.
Repo Etiquette
Branch naming (feat/, fix/), commit message format (conventional commits), PR conventions.

A good example CLAUDE.md

# Code style
- Use ES modules (import/export) syntax, not CommonJS (require)
- Destructure imports when possible (eg. import { foo } from 'bar')
- Use 2-space indentation
- Prefer const over let; never use var

# Build & Test
- Build: `npm run build`
- Lint: `npm run lint`
- Test all: `npm test`
- Test single: `npx jest path/to/test.ts`
- Typecheck: `npx tsc --noEmit`

# Workflow
- Be sure to typecheck when you're done making a series of code changes
- Prefer running single tests, not the whole suite, for performance
- Use conventional commits: feat:, fix:, docs:, refactor:, test:
- Branch names: feat/short-description or fix/short-description

# Architecture
- Handlers in src/api/handlers/ (one per resource)
- Business logic in src/services/ (handlers call services, never DB directly)
- Database models in src/models/ (Prisma)
- Shared utilities in src/utils/

# Gotchas
- AUTH_TOKEN_CONFIG env var must be set or token validation crashes
- Run `docker compose up -d` before integration tests (needs PostgreSQL)
- The CI build uses Node 20; don't use Node 22 features

What to Exclude Intermediate

Every line in CLAUDE.md consumes context tokens every session. Include only what changes Claude's behavior. If removing a line wouldn't cause Claude to make mistakes, delete it.

Do Not Include Why
Standard language conventions Claude already knows Claude knows Python PEP 8, JavaScript best practices, etc.
Things Claude can figure out by reading code It will read your files and infer patterns
Detailed API documentation Link to docs instead; or use a skill for on-demand loading
Information that changes frequently Stale instructions cause confusion
Long explanations or tutorials Be concise; use bullets, not paragraphs
File-by-file descriptions of the codebase Claude can read the files directly
Self-evident practices like "write clean code" Too vague to change behavior
Anti-Pattern

If your CLAUDE.md is too long, Claude ignores half of it because important rules get lost in the noise. If Claude keeps doing something you don't want despite having a rule against it, the file is probably too long. Prune ruthlessly.

Keeping It Lean Intermediate

Target under 500 lines per CLAUDE.md file (the official docs recommend under 200 for best adherence). Longer files consume more context and reduce how reliably Claude follows your rules.

Strategies for staying lean

  1. Use @path imports to pull in external files instead of inlining long content. The imported file is expanded at load time.
  2. Move specialized workflows to skills in .claude/skills/. Skills load on demand instead of every session.
  3. Use .claude/rules/ for path-scoped rules that only load when Claude works with matching files.
  4. Review regularly. Ask yourself for each line: "Would removing this cause Claude to make mistakes?" If not, cut it.

Import syntax

# In your CLAUDE.md
See @README.md for project overview and @package.json for npm commands.

# Additional Instructions
- Git workflow: @docs/git-instructions.md
- Personal overrides: @~/.claude/my-project-instructions.md

Both relative and absolute paths work. Relative paths resolve from the file containing the import. Imports can nest up to 5 levels deep.

Skills vs. CLAUDE.md

CLAUDE.md is loaded every session. Skills load on demand. If a block of instructions only applies when doing a specific workflow (like PR reviews or database migrations), move it to a skill. This keeps your base context smaller and improves adherence for the rules that remain.

Using Emphasis Intermediate

Claude responds to emphasis keywords. When a rule is critical and must never be violated, use strong language:

  • IMPORTANT: signals a high-priority rule
  • ALWAYS: for rules that must apply in every case
  • NEVER: for hard prohibitions
  • YOU MUST: for mandatory behaviors
# Critical rules
- IMPORTANT: Never commit directly to main. Always use feature branches.
- ALWAYS run `npm run typecheck` before committing TypeScript changes.
- NEVER use `any` type in TypeScript. Use `unknown` and narrow with type guards.
- YOU MUST include tests for any new API endpoint.
Use Sparingly

If every rule uses IMPORTANT and NEVER, none of them stand out. Reserve emphasis for the 3-5 rules that actually matter most. Use normal language for everything else.

The /init Command Beginner

You don't have to write CLAUDE.md from scratch. The /init command analyzes your codebase and generates a starting file with build commands, test instructions, and project conventions it discovers.

Terminal — /init Walkthrough
$ cd ~/my-project $ claude Starting Claude Code... claude> /init Analyzing your project... Detected: TypeScript, Node.js, Express, Prisma, Jest Found: package.json, tsconfig.json, .eslintrc, jest.config.ts Reading existing patterns... Generated CLAUDE.md with: - Build command: npm run build - Test command: npx jest - Lint command: npm run lint - Code style: ES modules, 2-space indent - Architecture: handler/service/model layers Review and refine CLAUDE.md with project-specific rules that Claude couldn't discover on its own.

If a CLAUDE.md already exists, /init suggests improvements rather than overwriting it.

Interactive Init

Set CLAUDE_CODE_NEW_INIT=true to enable the interactive multi-phase flow. This version asks which artifacts to set up (CLAUDE.md, skills, hooks), explores your codebase with a subagent, fills in gaps via follow-up questions, and presents a reviewable proposal before writing any files.

After /init

The generated file is a starting point. Refine it by adding:

  • Rules Claude wouldn't discover on its own (non-obvious gotchas, required environment variables)
  • Commit message and branch naming conventions
  • Architecture decisions that aren't obvious from the code structure
  • Testing preferences (prefer running single tests, avoid mocks, etc.)

Check CLAUDE.md into git so your team benefits. The file compounds in value over time as you add rules based on mistakes Claude makes.

Full annotated CLAUDE.md example 
# Acme API — CLAUDE.md

# Code Style
- Use ES modules (import/export), not CommonJS (require)
- Destructure imports: import { foo } from 'bar'
- 2-space indentation, single quotes, trailing commas
- Prefer const over let; NEVER use var
- ALWAYS use explicit return types on exported functions

# Build & Test
- Build: `npm run build`
- Lint: `npm run lint -- --fix`
- Typecheck: `npx tsc --noEmit`
- Test all: `npm test`
- Test single file: `npx jest path/to/file.test.ts`
- Test single test: `npx jest -t "test name pattern"`
- IMPORTANT: Run typecheck after any TypeScript changes

# Workflow
- Use conventional commits: feat:, fix:, docs:, refactor:, test:, chore:
- Branch names: feat/short-desc or fix/short-desc
- NEVER commit directly to main
- Prefer running single tests during development for speed
- Run the full suite before committing

# Architecture
- Route handlers: src/api/handlers/ (one file per resource)
- Business logic: src/services/ (handlers call services, never DB directly)
- Database: src/models/ (Prisma schema + generated client)
- Utilities: src/utils/ (pure functions, no side effects)
- Middleware: src/middleware/ (auth, rate-limiting, logging)
- New API routes go in src/api/routes.ts

# Testing
- Co-locate test files: foo.ts has foo.test.ts in the same directory
- Use real database for integration tests (not mocks)
- Factory functions for test data in tests/factories/
- IMPORTANT: Every new API endpoint must have at least one integration test

# Gotchas
- AUTH_TOKEN_CONFIG env var MUST be set or token validation crashes
- Run `docker compose up -d` before integration tests (needs PostgreSQL + Redis)
- CI uses Node 20 — do not use Node 22 features
- The /health endpoint is excluded from rate limiting (see middleware/rateLimit.ts)
- Prisma migrations: `npx prisma migrate dev` (never `migrate reset` in shared envs)

# External docs
- API design guide: @docs/api-conventions.md
- Deployment runbook: @docs/deploy.md

This example is about 45 lines — concise, scannable, and focused on things Claude cannot infer from code alone. Each section uses bullets for quick parsing. Critical rules use IMPORTANT and NEVER for emphasis.
