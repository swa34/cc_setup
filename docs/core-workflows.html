Core Workflows

The everyday patterns that make Claude Code productive: reviewing code, committing changes, debugging errors, and exploring unfamiliar codebases.

Code Review Beginner

Claude Code can review your changes before you commit, catching bugs, style issues, and logic errors. The simplest approach is the /review slash command, which examines your current diff and posts inline feedback.

Key Concept

For the most unbiased review, use a separate Claude session from the one that wrote the code. Claude won't be biased toward code it just produced. This is the Writer/Reviewer pattern.

Quick review with /review

Run /review inside a Claude session to review your uncommitted changes. Claude reads the diff, identifies potential issues, and explains its findings.

Terminal — Code Review
claude> /review Reviewing uncommitted changes... src/auth/session.ts Line 42: Token expiry check uses < instead of <=. A token expiring at exactly the boundary will pass validation but fail on the next API call. Consider using <= for the comparison. src/api/handlers/login.ts No issues found. src/middleware/rateLimit.ts Line 18: The rate limit counter resets on every deployment because it's stored in-memory. Consider using Redis or a shared store if you run multiple instances.

The Writer/Reviewer pattern

For high-stakes changes, use two separate sessions. Session A writes the code; Session B reviews it with fresh context and no bias toward the implementation.

Terminal — Session B (Reviewer)
$ claude Starting new session... claude> Review the rate limiter implementation in @src/middleware/rateLimit.ts. Look for edge cases, race conditions, and consistency with our existing middleware patterns. I'll review the rate limiter implementation... Findings: 1. Race condition: concurrent requests can exceed the limit because the read-then-increment is not atomic. Use INCR in Redis or a mutex for in-memory counters. 2. Missing X-RateLimit-Remaining header — other middleware in this project (see cors.ts, auth.ts) always set response headers. 3. Good: sliding window approach matches the pattern in throttle.ts.

Using /simplify for cleaner code

After a feature is working, run the /simplify prompt pattern to reduce complexity. Ask Claude to look for unnecessary abstractions, redundant code, or over-engineering.

Tip

You can also set up automated PR reviews with Claude's Code Review feature for GitHub. It uses a fleet of specialized agents that analyze your PR in context of the full codebase. See the Code Review docs for setup.

Committing & Pull Requests Beginner

Claude can create commits and open pull requests directly. Just tell it what you want. Claude examines your staged and unstaged changes, writes a descriptive commit message, and can push and create a PR in one go.

Creating commits

After making changes, ask Claude to commit. Claude analyzes the diff and writes a message following your project's conventions (which you can specify in CLAUDE.md).

Terminal — Committing
claude> commit my changes with a descriptive message I'll review your changes and create a commit. Changes to commit: modified: src/auth/session.ts modified: src/middleware/rateLimit.ts new file: src/tests/rateLimit.test.ts Created commit: fix(auth): use atomic counter for rate limiting Replace in-memory read-then-increment with atomic INCR to prevent race conditions under concurrent requests. Add X-RateLimit-Remaining header for consistency with existing middleware. Fix off-by-one in token expiry check.

Opening pull requests

Claude uses the gh CLI under the hood. Ask it to create a PR and it will push your branch, generate a title and description from the commit history, and open the PR on GitHub.

Terminal — Pull Request
claude> create a pr for my changes Pushing branch fix/rate-limit-atomicity to origin... Branch pushed. Creating pull request... PR #247 created: fix(auth): atomic rate limiting with race condition fix https://github.com/acme/app/pull/247
Commit Message Conventions

Add your commit message format to CLAUDE.md so Claude follows it automatically. For example: "Use conventional commits (feat:, fix:, docs:, etc). Keep the subject line under 72 characters."

Resuming from a PR

When you create a PR with Claude, the session is automatically linked. You can resume it later:

# Resume the session that created PR #247
claude --from-pr 247

This restores the full conversation context from when you created the PR, so you can address review feedback without re-explaining the background.

Debugging Intermediate

Claude excels at debugging because it can read your code, run commands, observe output, and iterate, all in one loop. The key is to give Claude the symptom and let it find the root cause, rather than pointing it at what you think the problem is.

Paste the error, let Claude investigate

The fastest way to debug is to paste the error output directly. Claude traces the stack, reads the relevant files, and proposes a fix.

Terminal — Debugging
claude> I'm seeing this error when I run npm test: TypeError: Cannot read properties of undefined (reading 'refresh') at SessionManager.validateToken (src/auth/session.ts:42:28) at LoginHandler.handle (src/api/handlers/login.ts:18:35) Let me investigate the error... Reading src/auth/session.ts... Reading src/api/handlers/login.ts... Root cause found: In session.ts line 38, `getTokenConfig()` can return undefined when the environment variable AUTH_TOKEN_CONFIG is not set. Line 42 then tries to access `.refresh` on that undefined value. This isn't a test issue — it's a missing null check that would also crash in production if the env var is unset. Fix: I'll add a null check with a sensible default, and a test that covers the missing-config case.
Root Cause vs. Symptoms

Tell Claude to address the root cause, not suppress the error. A common failure mode: Claude adds a try/catch that swallows the exception instead of fixing the underlying issue. Be explicit: "Fix the root cause. Don't suppress the error."

The debugging loop

For trickier bugs, Claude works through an iterative cycle: reproduce, hypothesize, test, verify. You can guide this process or let Claude drive.

  1. Share the error and reproduction steps — paste the output, mention if it's intermittent
  2. Let Claude investigate — it reads files, checks git history, runs commands
  3. Review the proposed fix — Claude explains its reasoning before changing code
  4. Verify — Claude runs tests to confirm the fix works
Example debugging prompts

When you have a stack trace:

# Paste the error and let Claude trace it
I'm seeing this error when I run npm test:
[paste full stack trace]
Fix the root cause, don't suppress the error.

When behavior is wrong but no error:

# Describe expected vs actual behavior
Users report that login fails after session timeout.
Check the auth flow in src/auth/, especially token refresh.
Write a failing test that reproduces the issue, then fix it.

When the bug is intermittent:

# Give Claude reproduction context
This test fails about 30% of the time in CI:
tests/integration/checkout.test.ts
It always passes locally. Suspect a timing issue.
Investigate and fix it.

When you want a targeted fix:

# Point Claude to the area
suggest a few ways to fix the @ts-ignore on line 15 of user.ts

Exploring Codebases Intermediate

When you're new to a codebase or investigating unfamiliar code, Claude's Plan Mode is your best tool. It lets Claude analyze code read-only, so it explores thoroughly without changing anything.

Plan Mode (Shift+Tab)

Press Shift+Tab to cycle through permission modes. From Normal Mode, the first press enables Auto-Accept (edits auto-approved); the second press activates Plan Mode, indicated by ⏸ plan mode on at the bottom of the terminal.

In Plan Mode, Claude uses read-only operations: it reads files, searches code, and runs safe commands, but never writes or modifies. This is perfect for exploration and planning.

Terminal — Plan Mode Exploration
claude> give me an overview of this codebase ⏸ plan mode on Reading project structure... Reading package.json, tsconfig.json... Scanning src/ directory... Project Overview: This is a Node.js REST API built with Express and TypeScript. Key directories: src/api/handlers/ - Route handlers (one per resource) src/auth/ - Session management, JWT utilities src/middleware/ - Express middleware (auth, rate limiting, CORS) src/models/ - Prisma database models src/services/ - Business logic layer Patterns: - Handler -> Service -> Model layering - JWT-based auth with refresh tokens - Prisma ORM with PostgreSQL - Jest for testing, co-located test files

Start broad, then drill down

The most effective exploration pattern is to start with high-level questions and narrow progressively. Claude remembers earlier answers in the same session, so each follow-up builds on previous context.

Step 1: Overview
"Give me an overview of this codebase" or "Explain the main architecture patterns"
Step 2: Subsystem
"How does authentication work?" or "Trace the login flow from front-end to database"
Step 3: Specifics
"What edge cases does SessionManager handle?" or "Why does line 134 call foo() instead of bar()?"

Starting in Plan Mode from the command line

You can also launch Claude directly in Plan Mode:

# Start an interactive session in Plan Mode
claude --permission-mode plan

# Run a one-off analysis in Plan Mode (headless)
claude --permission-mode plan -p "Analyze the authentication system and suggest improvements"
Tip

Press Ctrl+G to open Claude's plan in your default text editor, where you can edit it before Claude proceeds. Useful for refining the approach before switching to Normal Mode for implementation.

Useful prompts for codebase exploration

Understanding structure:

  • give me an overview of this codebase
  • explain the main architecture patterns used here
  • what are the key data models?
  • how is authentication handled?
  • what's the testing strategy?

Finding code:

  • find the files that handle user authentication
  • how do these authentication files work together?
  • trace the login process from front-end to database
  • where are the database migrations?

Understanding decisions:

  • why does this code call foo() instead of bar() on line 333?
  • what edge cases does CustomerOnboardingFlowImpl handle?
  • look through ExecutionFactory's git history and summarize how its API came to be

Planning changes:

  • I need to add Google OAuth. What files need to change? What's the session flow? Create a plan.
  • how would I add a new API endpoint following this project's patterns?
  • what would break if I renamed the User model to Account?
When to Skip Planning

Plan Mode adds overhead. For tasks where the scope is clear and the fix is small (a typo, a log line, renaming a variable), skip the plan and ask Claude to just do it. Planning is most valuable when you're uncertain about the approach, the change modifies many files, or you're unfamiliar with the code.
