Working with Legacy Codebases

Strategies for navigating, understanding, and modernizing large, unfamiliar repositories using Claude Code as your guide.

The Challenge Intermediate

Legacy codebases are fundamentally different from greenfield projects. The code works — often in production for years — but understanding why it works and how to change it safely is the hard part.

No Documentation
The README was last updated three years ago. Architecture decisions live in the heads of engineers who have since left the company. Comments, if they exist, describe what the code does, not why.
Tangled Dependencies
Modules import each other in circular patterns. A change in one file cascades through dozens of others. The dependency graph looks less like a tree and more like a plate of spaghetti.
Unfamiliar Patterns
The codebase uses patterns from a different era or framework version. What looks "wrong" might be intentional. What looks intentional might be a workaround for a long-fixed bug.
Tribal Knowledge
"Don't touch the billing module" and "always restart after changing configs" are rules that live in Slack threads and standup notes, not in any file the code can see.

Claude Code is uniquely suited for these challenges. It can read hundreds of files, trace execution paths, analyze git history, and explain patterns — all the things that make onboarding to a legacy codebase so painful when done manually.

Onboarding Strategy Intermediate

Use this four-step process when you first encounter a legacy codebase. Each step builds on the previous one:

Step 1: Get the overview

Start broad. Let Claude survey the project structure, identify the tech stack, and give you a map of the territory.

Terminal
$ cd ~/work/legacy-monolith $ claude claude> Give me an overview of this codebase. What's the tech stack, how is it organized, and what are the main entry points? This is a Java/Spring Boot monolith with ~450 source files... Main entry points: Application.java, scheduled jobs in /jobs/, REST controllers in /api/...

Step 2: Understand the architecture

Once you have the overview, dig into how the pieces fit together. Ask about layers, data flow, and external integrations.

Terminal
claude> Explain the main architecture patterns used here. How do requests flow from the API layer to the database? The app follows a layered architecture: Controllers -> Services -> Repositories... claude> What external services does this connect to? Check configuration files and service classes. Found 5 external integrations: Stripe (payments), SendGrid (email), S3 (files)...

Step 3: Trace key flows

Pick the most important user journeys and trace them end-to-end. This reveals how the abstractions actually work in practice.

Terminal
claude> Trace the user login flow from the HTTP request to the session being created. Include every file involved. Login flow: LoginController.login() -> AuthService.authenticate() -> UserRepository.findByEmail()... claude> Trace how a payment is processed. Start from the checkout endpoint and follow through to Stripe. Payment flow: CheckoutController -> OrderService.createOrder() -> PaymentService.charge()...

Step 4: Learn the conventions

Every codebase has unwritten rules. Ask Claude to identify them so you can follow the same patterns when making changes.

Terminal
claude> What naming conventions does this codebase use? Look at classes, methods, database tables, and API endpoints. Classes: PascalCase with Impl suffix for implementations (e.g., PaymentServiceImpl)... claude> How are errors handled in this codebase? Is there a consistent pattern? Error handling uses custom exception hierarchy: BaseException -> BusinessException, TechnicalException...

Claude as Codebase Guide Intermediate

Think of Claude as the senior engineer you wish you could pair with on day one. Ask it the questions you would ask a colleague who built the system:

Understanding systems

Terminal
claude> How does logging work in this project? Where are logs configured and what levels are used? Logging uses SLF4J with Logback. Config at src/main/resources/logback-spring.xml... claude> How do I make a new API endpoint? Show me the pattern by looking at existing endpoints. Based on existing endpoints, the pattern is: 1) Create a DTO in /dto/ 2) Add a method to the service... claude> What edge cases does the OrderService handle? Look at the implementation and tests. OrderService handles: partial payments, out-of-stock during checkout, concurrent order creation...

Understanding history

Terminal
claude> Look through the git history of PaymentService and summarize how its API came to be. Why is it shaped the way it is? PaymentService evolved over 3 phases: 1) Single payment method (2021) 2) Multi-provider support added (2022)... claude> Why does this code call retryWithBackoff() instead of a simple retry? Check the git blame and PR history. Introduced in commit abc123 after a production incident. The PR mentions that Stripe returns 429s during peak hours...
Pro Tip

If you have the gh CLI installed, Claude can read PR descriptions, review comments, and issue discussions to understand why code was written, not just what it does.

Plan Mode Workflow Intermediate

When making changes to legacy code, never jump straight to implementation. Use the Explore-Plan-Implement-Commit cycle to minimize risk:

Explore
Read & understand
→
Plan
Design changes
→
Implement
Write & test
→
Commit
Review & push

Phase 1: Explore (Plan Mode)

Enter Plan Mode with Shift+Tab. Claude reads files and answers questions without making any changes.

Terminal — Plan Mode
claude (plan)> Read src/auth/ and understand how we handle sessions and login. Also look at how we manage environment variables for secrets. The auth module uses JWT tokens stored in HTTP-only cookies. Session management... Environment variables are loaded via dotenv from .env.local, with defaults in config/defaults.js...

Phase 2: Plan (Plan Mode)

Ask Claude to create a detailed implementation plan. Review it before any code gets written.

Terminal — Plan Mode
claude (plan)> I want to add Google OAuth. What files need to change? What's the session flow? Create a plan. Plan: Add Google OAuth Support 1. Add google-auth-library to package.json 2. Create src/auth/providers/google.js (new file) 3. Modify src/auth/AuthService.js to support multiple providers... 4. Add OAuth callback route in src/routes/auth.js... 5. Update .env.example with GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET...
Editing the Plan

Press Ctrl+G to open the plan in your text editor for direct editing before Claude proceeds. This lets you add constraints, remove steps, or reorder the approach.

Phase 3: Implement (Normal Mode)

Switch back to Normal Mode and let Claude execute the plan. Always include a verification step.

Terminal — Normal Mode
claude> Implement the OAuth flow from your plan. Write tests for the callback handler, run the test suite and fix any failures. Creating src/auth/providers/google.js... Modifying AuthService.js to support provider-based auth... Running tests... 2 failures found. Fixing... All 47 tests passing.

Phase 4: Commit

Terminal
claude> Commit with a descriptive message and open a PR Committing: "feat: add Google OAuth provider with session support" PR #142 created: https://github.com/org/repo/pull/142

Fan-Out Patterns Advanced

When legacy modernization involves changing dozens or hundreds of files, you need strategies that scale beyond one-at-a-time editing.

/batch for parallel changes

The /batch skill orchestrates large-scale changes across a codebase. It researches the scope, decomposes work into independent units, and spawns one agent per unit in an isolated git worktree:

Terminal
claude> /batch migrate all React class components in src/components/ to functional components with hooks Analyzing scope... Found 23 class components to migrate. Decomposing into 23 independent units... Spawning agents in isolated worktrees... Progress: [==================] 23/23 complete 23 PRs created for review.

Scripting with claude -p

For custom migration scripts, use claude -p in a loop. This is useful when each file needs the same transformation:

#!/bin/bash
# migrate-to-typescript.sh
# Convert JavaScript files to TypeScript one at a time

# Generate the file list
claude -p "List all .js files in src/ that should be converted to TypeScript. \
  Skip test files and config files. Output one file path per line, nothing else." \
  > files-to-migrate.txt

# Migrate each file
for file in $(cat files-to-migrate.txt); do
  echo "Migrating $file..."
  claude -p "Convert $file from JavaScript to TypeScript. \
    Add proper type annotations. \
    Rename the file from .js to .ts. \
    Fix any imports in other files that reference this file. \
    Run tsc --noEmit to verify there are no type errors." \
    --allowedTools "Read,Write,Edit,Bash(tsc *),Bash(mv *)"
done

echo "Migration complete. Review changes with: git diff"

Worktrees for isolation

When you need to make changes that might conflict, use git worktrees to work in parallel branches:

# Create isolated worktrees for different migration tasks
git worktree add ../project-auth-migration feature/auth-migration
git worktree add ../project-db-migration feature/db-migration

# Run Claude in each worktree simultaneously
cd ../project-auth-migration && claude -p "Migrate auth from sessions to JWT" &
cd ../project-db-migration && claude -p "Migrate from MongoDB to PostgreSQL" &

wait
echo "Both migrations complete. Review separately, merge sequentially."
Caution

Always test on a few files first before running at scale. Refine your prompt based on what goes wrong with the first 2-3 files. The --allowedTools flag restricts what Claude can do, which matters when running unattended.

Context Strategies Advanced

Legacy codebases are context-hungry. A single debugging session can read dozens of files and fill the context window fast. These strategies keep you productive:

Work in focused areas

Do not try to understand the entire codebase in one session. Focus on one module, one feature, or one flow at a time. Use /clear between unrelated investigations.

Use subagents for research

When Claude needs to explore the codebase to answer a question, delegate to a subagent. The subagent runs in a separate context window and reports back a summary, keeping your main conversation clean:

Terminal
claude> Use subagents to investigate how our authentication system handles token refresh, and whether we have any existing OAuth utilities I should reuse. [Subagent exploring auth module...] Summary: Token refresh is handled by RefreshTokenService. It uses a rotating refresh token pattern... Found existing OAuthClient utility in src/lib/oauth.js that handles provider registration...

Start fresh sessions between tasks

If you have corrected Claude more than twice on the same issue, the context is cluttered with failed approaches. Run /clear and start fresh with a more specific prompt that incorporates what you learned. A clean session with a better prompt almost always outperforms a long session with accumulated corrections.

Use /compact strategically

When auto-compaction triggers, Claude summarizes what matters most. You can guide this by adding instructions to CLAUDE.md:

# In your CLAUDE.md
When compacting, always preserve:
- The full list of modified files
- Any test commands that were run
- Architecture decisions made during the session
- The current task state and next steps

Building CLAUDE.md Incrementally Intermediate

For legacy codebases, do not try to write a perfect CLAUDE.md on day one. Build it incrementally as you learn the codebase. Treat it as living documentation that grows with your understanding.

Start with /init

Run /init in the project root. Claude analyzes the codebase and generates a starting CLAUDE.md with build commands, test instructions, and conventions it discovers:

Terminal
$ cd ~/work/legacy-monolith $ claude claude> /init Analyzing your project... Detected: Java 17, Spring Boot 3.x, Gradle, PostgreSQL, Redis Found: 12 test suites, CI config in .github/workflows/ Generated CLAUDE.md with project context

Add as you learn

Every time you discover something non-obvious, add it to CLAUDE.md. Here is a growth pattern over the first week:

# CLAUDE.md — built incrementally

# Build & Test
- `./gradlew build` to compile (takes ~2 min)
- `./gradlew test` for unit tests
- `./gradlew integrationTest` requires local PostgreSQL on port 5433 (not 5432!)
- Redis must be running on port 6380 for session tests

# Architecture
- Layered: Controllers -> Services (with Impl suffix) -> Repositories
- DTOs in /dto/, entities in /model/, mappers in /mapper/
- All API responses wrapped in ApiResponse<T> envelope

# Conventions
- Exception hierarchy: BaseException -> BusinessException, TechnicalException
- Logging: SLF4J with Logback, config at src/main/resources/logback-spring.xml
- DB migrations: Flyway in src/main/resources/db/migration/ (V{n}__{desc}.sql)

# Gotchas
- PaymentService retries with exponential backoff due to Stripe 429s during peak
- Don't modify OrderStatus enum order — Hibernate maps by ordinal, not name
- The "legacy" package contains code shared with the old monolith — do not refactor

Keep it under 500 lines

CLAUDE.md is loaded every session. If it grows beyond ~500 lines, Claude starts ignoring instructions because important rules get lost in the noise. Strategies to keep it lean:

  • Move detailed content into .claude/rules/ files scoped to specific directories
  • Use @imports for reference material: @docs/api-guide.md
  • Prune regularly — if Claude already does something correctly without the instruction, delete it
  • Convert patterns to hooks — if a rule must always be followed, make it a hook instead of an instruction
Team Collaboration

Check CLAUDE.md into git so your team can contribute. Each developer adds the gotchas and conventions they discover. Over time, it becomes the documentation the codebase never had — written in a format optimized for an AI pair programmer.
