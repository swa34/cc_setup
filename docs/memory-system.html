Memory System

How Claude Code remembers your project across sessions — from static instructions to automatic note-taking and structured memory hierarchies.

Memory Types Overview Intermediate

Each Claude Code session starts with a fresh context window. Two built-in mechanisms carry knowledge across sessions, and you can add a third with a structured memory system:

CLAUDE.md (Static Context)

Instructions you write that Claude reads at the start of every session. Think of it as your project's operating manual for Claude.

Who writes it
You (the developer)
Scope
Project, user, or organization
Contains
Coding standards, build commands, architecture decisions
Loaded
Fully, at every session start
Auto Memory (Claude's Notes)

Notes Claude writes itself as it discovers patterns, preferences, and project details during your sessions.

Who writes it
Claude (automatically)
Scope
Per working tree / git repo
Contains
Build insights, debugging patterns, workflow habits
Loaded
First 200 lines of MEMORY.md at session start
Custom Memory Files

A structured system you design using indexed markdown files organized by topic, domain, and tool.

Who writes it
You + Claude (collaborative)
Scope
Cross-project or project-specific
Contains
Domain knowledge, tool configs, conventions
Loaded
On demand via @imports or Claude reading files
Key Concept

CLAUDE.md and Auto Memory are complementary. CLAUDE.md is what you tell Claude; Auto Memory is what Claude tells itself. Both are loaded at every session start. Use CLAUDE.md for standards and rules; let Auto Memory capture the rest.

Auto Memory Intermediate Docs

Auto Memory lets Claude accumulate knowledge across sessions without you writing anything. As Claude works, it saves notes for itself: build commands that work, debugging insights, architecture patterns, and your coding preferences. Claude decides what is worth remembering based on whether it would be useful in a future conversation.

Auto Memory is on by default (requires Claude Code v2.1.59+). Toggle it with the /memory command or in your project settings:

{
  "autoMemoryEnabled": false
}

Where auto memory lives

Each project gets its own memory directory derived from the git repository path. All worktrees and subdirectories within the same repo share one auto memory directory:

# Auto memory directory structure
~/.claude/projects/<project>/memory/
├── MEMORY.md              # Concise index — loaded every session (first 200 lines)
├── debugging.md           # Detailed notes on debugging patterns
├── api-conventions.md     # API design decisions Claude discovered
└── build-system.md        # Build commands and workarounds
200-Line Limit

Only the first 200 lines of MEMORY.md are loaded at session start. Content beyond line 200 is invisible until Claude explicitly reads the file. Claude keeps MEMORY.md concise by moving detailed notes into separate topic files.

How it works in practice

When you see "Writing memory" or "Recalled memory" in the Claude Code interface, Claude is actively updating or reading from the memory directory. Claude does not save something every session — it decides what is worth remembering based on whether the information would be useful in a future conversation.

Topic files like debugging.md or patterns.md are not loaded at startup. Claude reads them on demand using its standard file tools when it needs the information.

Tip

Run /memory to browse and edit auto memory files. Everything is plain markdown you can edit or delete at any time. When you ask Claude to "remember" something (like "always use pnpm, not npm"), it saves to auto memory.

Setting Up Persistent Memory Intermediate

Beyond the built-in Auto Memory, you can create a structured memory system that gives you full control over what Claude remembers. This is especially useful when you want cross-project knowledge, domain-specific references, or tool configurations that persist across all your work.

The structured memory system uses an indexed hierarchy of markdown files rooted at ~/.claude/memory/:

# Structured memory directory
~/.claude/memory/
├── memory.md                  # Index of all memory files (always read first)
├── general.md                 # Cross-project facts, preferences, environment
├── domain/
│   ├── kubernetes.md          # Domain: container orchestration patterns
│   ├── graphql.md             # Domain: GraphQL schema design
│   └── auth-patterns.md       # Domain: authentication and authorization
└── tools/
    ├── docker.md              # Tool: Docker CLI patterns and workarounds
    ├── git.md                 # Tool: Git workflows and aliases
    └── claude-code.md         # Tool: Claude Code tips and configuration

The memory.md index

The memory.md file serves as a table of contents. Claude reads it at session start and loads other files only when they become relevant to the current task:

# Memory Index

## General
- general.md — cross-project conventions, editor prefs, env setup

## Domain Knowledge
- domain/kubernetes.md — k8s patterns, helm charts, debugging pods
- domain/graphql.md — schema design, resolver patterns, N+1 prevention
- domain/auth-patterns.md — OAuth flows, JWT handling, session management

## Tool Configs
- tools/docker.md — multi-stage builds, compose patterns, troubleshooting
- tools/git.md — branching strategy, rebase workflow, hooks
- tools/claude-code.md — prompt patterns, context management, cost tips
Pro Tip

For a full walkthrough of setting up the structured memory system with auto-init and maintenance commands, see the Memory Setup Guide.

Organization Patterns Intermediate

The key to effective memory is organizing by topic, not chronology. Each memory entry should answer: "What did I learn, and when might I need it again?"

Good memory entries

Each entry should include a date, what was learned, and why it matters — nothing more:

# domain/kubernetes.md

## 2025-01-15 — Pod health checks need initialDelaySeconds
Discovered that our Spring Boot apps take 30-45s to start.
Without initialDelaySeconds: 60 on liveness probes, k8s kills
pods during rolling deployments. Set in all deployment manifests.

## 2025-02-03 — HPA scaling threshold should be 70% not 80%
At 80% CPU target, we hit OOM before scale-up completes.
Dropped to 70% after prod incident on payment-service.

## 2025-03-10 — Use topologySpreadConstraints over podAntiAffinity
podAntiAffinity is too strict for our 3-node staging cluster.
topologySpreadConstraints with maxSkew: 1 gives better distribution.

Bad memory entries (avoid these)

# BAD: Too vague, no actionable detail
## Fixed a bug
There was a bug in the code and I fixed it.

# BAD: Too verbose, buries the lesson
## Long debugging session
First I tried X, then Y, then Z, then I read the docs,
then I asked on Slack, then I found that...
[20 more lines of narrative]

# BAD: Duplicates what CLAUDE.md should hold
## Code style
Use 2-space indentation. (This belongs in CLAUDE.md, not memory.)
Rule of Thumb

CLAUDE.md = rules Claude should always follow. Memory files = knowledge Claude should recall when relevant. If removing an entry from CLAUDE.md would cause Claude to make mistakes, it belongs in CLAUDE.md. If it is background knowledge that informs decisions, it belongs in memory.

Splitting files that grow too large

When a topic file exceeds ~100 entries, split it. For example, domain/kubernetes.md might split into:

domain/
├── k8s-networking.md      # Services, ingress, DNS
├── k8s-storage.md         # PVCs, storage classes, CSI
├── k8s-scaling.md         # HPA, VPA, cluster autoscaler
└── k8s-debugging.md       # Pod troubleshooting, log patterns

Update memory.md to reflect the new structure so Claude can find what it needs.

Cross-Session Knowledge Intermediate

Getting Claude to remember project context across sessions requires intentional design. Here is what to persist and what to leave out:

Persist (worth remembering)Skip (do not persist)
Build commands that differ from defaults Temporary workarounds already fixed
Architecture decisions and their rationale Step-by-step debugging logs
Non-obvious project conventions Information Claude can infer from code
Environment quirks (required env vars, port conflicts) Standard language conventions
Integration gotchas (API rate limits, auth flows) Obvious patterns ("use async/await")
Domain-specific terminology and business rules Code that is self-documenting

Making Claude remember specific things

You can explicitly ask Claude to remember things during a session. Claude saves these to auto memory:

Terminal
claude> Remember that our API tests require a local Redis instance on port 6380 (not the default 6379) I'll save that to memory. In future sessions, I'll know to use port 6380 for Redis in API tests. claude> Always use pnpm, not npm, in this project Noted. I'll use pnpm for all package management operations in this project.

For instructions that should go into CLAUDE.md rather than auto memory, be explicit: "Add to CLAUDE.md: always use pnpm". Or use /memory to open and edit files directly.

Project MEMORY.md for auto-init

You can create a project-specific MEMORY.md that gets initialized automatically when Claude first encounters the project. This gives Claude a head start with project-specific knowledge:

# MyProject - Project Memory

## Global Memory
Read ~/.claude/CLAUDE.md for memory rules and topic files.

## Project Notes
- Uses PostgreSQL 15 with pgvector extension for embeddings
- CI runs on GitHub Actions; deploy via ArgoCD
- Frontend is Next.js 14 with App Router (not Pages)
- API follows JSON:API spec with envelope wrapping

Troubleshooting Intermediate

When Claude ignores your instructions or forgets things between sessions, work through these checks:

Claude is not following CLAUDE.md instructions

Diagnostic checklist
  1. Verify it is loaded: Run /memory and confirm your CLAUDE.md appears in the list. If it is not listed, Claude cannot see it.
  2. Check the file location: CLAUDE.md must be in a location that gets loaded for your working directory (project root, .claude/CLAUDE.md, or ~/.claude/CLAUDE.md).
  3. Check file size: Target under 200 lines per CLAUDE.md. Longer files consume more context and reduce adherence. If yours is too large, split into .claude/rules/ files.
  4. Make instructions specific: "Use 2-space indentation" works better than "format code nicely."
  5. Check for conflicts: If two CLAUDE.md files give different guidance for the same behavior, Claude may pick one arbitrarily.
  6. Add emphasis for critical rules: Use IMPORTANT: or YOU MUST to improve adherence on rules that matter most.

Auto memory saved unexpected content

How to audit and clean up

Run /memory and select the auto memory folder to browse what Claude saved. Everything is plain markdown — edit or delete entries freely. You can also browse the directory directly:

# List all auto memory files for a project
ls ~/.claude/projects/<project>/memory/

# Read the MEMORY.md index
cat ~/.claude/projects/<project>/memory/MEMORY.md

# Delete a topic file if it contains bad info
rm ~/.claude/projects/<project>/memory/outdated-notes.md

Instructions lost after /compact

Why this happens and how to fix it

CLAUDE.md fully survives compaction. After /compact, Claude re-reads your CLAUDE.md from disk and re-injects it into the session. If an instruction disappeared after compaction, it was given only in conversation, not written to CLAUDE.md.

Fix: Add the instruction to CLAUDE.md so it persists across sessions and compaction. You can also use a SessionStart hook with a compact matcher to re-inject critical context:

{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Reminder: use pnpm, not npm. Run tests before committing.'"
          }
        ]
      }
    ]
  }
}

CLAUDE.md is too large

Solutions

If your CLAUDE.md exceeds 200 lines:

  • Move detailed content into separate files referenced with @path imports
  • Split instructions into .claude/rules/ files scoped to file patterns
  • Move domain knowledge to structured memory files that Claude reads on demand
  • Prune anything Claude already does correctly without the instruction
