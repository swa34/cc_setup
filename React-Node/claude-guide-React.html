The Big Picture — What Goes Where Docs

Claude Code has several layers of configuration. Here is how they all fit together:

Layer What it does When to use it
CLAUDE.md Static project context Claude reads every session Architecture, conventions, build commands
Memory Persistent notes Claude reads/writes across sessions User preferences, feedback, project context
Hooks Scripts/agents that fire automatically on events Post-commit tasks, validation, reminders
Skills Reusable slash commands you invoke manually Repeated workflows, templates, procedures
Plugins Packaged bundles of skills + hooks + MCP servers Shared toolkits (code review, testing, etc.)
MCP Servers External tool connections (APIs, databases, etc.) GitHub, Docker, Git, databases, browsers
Settings Permissions, env vars, plugin toggles What Claude can/can't do without asking

The key distinction

  • CLAUDE.md = "here's what you need to know"
  • Memory = "here's what you've learned"
  • Hooks = "when X happens, do Y automatically"
  • Skills = "when I say /X, do Y"
  • MCP = "here are external tools you can use"

CLAUDE.md — Project Instructions Docs

The main way to give Claude context about a project. Claude reads it at the start of every session.

Locations (most specific wins)

File Scope Shared
~/.claude/CLAUDE.md All your projects No
<repo>/CLAUDE.md One project Yes (committed)
<repo>/.claude/rules/*.md Path-scoped rules Yes

What to put in CLAUDE.md

Project overview — what it is, who it's for, what stage it's in:

Budget Contributions app for UGA CAES. ~753 county personnel enter annual budget data. Phase 4 (testing/polish) in progress.

Tech stack and architecture — so Claude doesn't guess wrong:

Node.js + Express 5 + TypeScript (ESM), PostgreSQL 17 via Knex 3.
Frontend: React 19 + Vite 7 + Tailwind 4. AG Grid for spreadsheet views.

Build / run / test commands — the exact commands, not generic ones:

docker compose up -d # Full stack
npm run migrate # Knex migrations
npm test # Vitest (backend: 143 tests)

What NOT to put in CLAUDE.md

Event-driven workflows (use hooks instead):

Bad: "After every commit, run the linter and check for TODO comments"
Good: PostToolUse hook on Bash that detects git commit and runs checks

Repeated manual procedures (use skills instead):

Bad: "When I say 'scaffold component', create files X, Y, Z..."
Good: /scaffold-component skill with templates

Rules directory (.claude/rules/)

When CLAUDE.md gets too long, split detailed rules into separate files. Each file covers one topic. Rules without a paths field load every session. Rules with paths only load when Claude works with matching files — saving context tokens.

Directory structure

your-project/
  .claude/
    CLAUDE.md              # Keep short — overview + build commands
    rules/
      testing.md           # Test conventions (loads every session)
      api-design.md        # API patterns (loads every session)
      frontend-styles.md   # CSS/styling (only when editing frontend files)
      security.md          # Security requirements (loads every session)

Example: rule that applies to all files

# Testing Rules

- Use Vitest for all tests
- Backend tests use supertest for HTTP assertions
- Never mock the database in integration tests
- Test files live in `src/__tests__/` not next to source files

Example: rule scoped to specific paths

---
paths:
  - 'frontend/src/components/**/*.tsx'
  - 'frontend/src/styles/**/*.css'
---

# Frontend Styling Rules

- Use Tailwind utility classes — no inline styles or CSS modules
- Use CSS custom properties from the @theme block
- Never hardcode hex colors or font names
- AG Grid overrides go in inventory-grid.css, not component styles

This rule only loads when Claude reads or edits files matching those glob patterns. Backend work won't waste tokens loading frontend rules.

Path pattern examples

Pattern Matches
src/**/*.test.ts All test files
frontend/**/*.tsx All React components
backend/src/routes/** All route files
**/*.css Any CSS file anywhere
src/api/**/*.ts API layer files

Memory — Persistence Across Sessions Docs

Claude Code has a built-in memory system so it remembers things across conversations. Memory is fully automatic — you don't invoke memory types or manage them manually. Claude decides on its own when something is worth saving and which type fits best.

How it works

  • Auto-memory: Claude writes notes to ~/.claude/projects/<project>/memory/MEMORY.md
  • Global CLAUDE.md: ~/.claude/CLAUDE.md holds rules that apply everywhere
  • First 200 lines of MEMORY.md are loaded each session; topic files loaded on demand.
  • You can also say "remember that..." to explicitly ask Claude to save something.

Memory types (automatic — Claude assigns these)

These are categories Claude uses internally when saving notes. You don't pick them — Claude chooses the right type based on what it learned. Each type shapes how the memory is written and when it gets recalled in future sessions.

Type What Claude saves When Claude saves it Example (county_cont)
user Your role, expertise, preferences When it learns who you are or how you work "UGA CAES developer, deep Express/Knex experience, new to AG Grid"
feedback Corrections and confirmed approaches When you correct Claude or confirm a non-obvious choice worked "Don't mock PostgreSQL in integration tests — use the Docker Compose db service. Mocked tests hid a broken Knex migration last quarter."
project Ongoing work, decisions, deadlines When it learns about goals, status, or why a decision was made "Inventory module attestation workflow on hold until Julia confirms the signature field requirements — 2026-03-25 deadline."
reference Pointers to external resources When you mention where info lives outside the codebase "Inventory bug reports tracked in the county_cont GitHub Issues board. UGA personnel API docs at caes-api.uga.edu/docs."

What you do vs. what Claude does

You Claude
Just work normally — correct mistakes, confirm approaches, mention deadlines Watches for patterns worth saving and writes them to memory files
Say "remember that funding_source IDs are string slugs, not UUIDs" Saves it immediately as a feedback memory
Say "forget that" or edit MEMORY.md directly Removes the entry or defers to your manual edits
Run /memory Shows all loaded CLAUDE.md and memory files for review
Say "reorganize memory" Deduplicates, merges, and cleans up memory files

Bootstrap the memory system

Use the Memory System Setup prompt to create the full directory structure, global index, CLAUDE.md rules, and per-project stubs in one shot. One-time setup — after that, Claude manages memory automatically.

Hooks — Automated Reactions Docs

Hooks fire automatically when specific events happen. They're the backbone of workflow automation.

When to use hooks

Use a hook when you want something to happen automatically. Good hook candidates:

  • "After every commit, run the linter and check for broken tests"
  • "Before stopping, verify all tests pass"

Hook events

Event When it fires Common use
PreToolUse Before a tool runs Block dangerous commands
PostToolUse After a tool completes Lint, format, post-commit tasks
Stop Before Claude stops Verify work is complete

Configuration Example

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/my-script.sh",
            "timeout": 5000
          }
        ]
      }
    ]
  }
}

Exit codes for Command Hooks

  • 0: Success — continue
  • 2: Block — stderr message shown to Claude, must respond
  • Other: Non-blocking warning

Skills — Reusable Workflows Docs

Skills are slash commands you invoke manually. They're for repeated workflows that need your trigger, not automatic reactions.

When to use skills

  • "Set up a new React component with tests and storybook"
  • "Review this PR with our team's checklist"
  • "Bootstrap a new Vite + React project with our standard stack"

Skill file format

Skills live in .claude/skills/ (or a plugin's skills/ directory). Each is a Markdown file with YAML frontmatter:

---
name: my-skill
description: >
  When Claude should suggest this skill. Include trigger phrases
  so Claude activates it at the right time.
---
Step-by-step instructions for what Claude should do.
Can reference files, run commands, use tools.

Key frontmatter fields

Field Purpose
name Unique identifier — used as the slash command name
description Trigger logic — Claude uses this to decide when to suggest the skill. Include keywords and phrases users might say.
disable-model-invocation Optional. If true, the skill only fires when you type /name — Claude won't auto-suggest it.

Using skills

Invoke via terminal inside a session:

/my-skill              # Invoke by name
/my-skill some args    # Pass arguments

Claude can also suggest a skill automatically if the description matches what you're asking about (unless disable-model-invocation: true is set).

Example: Vite + React Setup Skill

This project includes a skills/new-project-setup.md skill that provides opinionated guidance when bootstrapping new Vite + React applications. It activates during planning and design phases.

How it triggers

The skill's description includes trigger phrases like "new project", "scaffold", "setup", "what stack should I use", "project structure", and "Vite + React". When you ask about any of these topics, Claude loads the full skill and follows its guidance.

What the skill covers

Section Guidance
Core stack Vite + React + TypeScript (strict) + React Router v7+
State management React Context + TanStack Query for most apps; Zustand/Redux only when needed
Styling Tailwind CSS v4 (@theme tokens), optional shadcn/ui with design-system wrappers
Code quality ESLint + Prettier (existing teams) or Biome (greenfield)
Testing Vitest + Testing Library + Supertest + Playwright
Project structure Feature-based architecture with features/ and shared/ directories
Backend Node.js + Express 5 + TypeScript (ESM) + Knex + PostgreSQL
Dev environment Docker Compose, Vite proxy config, path aliases

Feature-based structure (from the skill)

The skill recommends organizing by domain feature, not by file type:

src/
├── features/              # Feature modules
│   ├── auth/
│   │   ├── components/    # Feature-specific UI
│   │   ├── hooks/         # Feature-specific hooks
│   │   ├── api/           # Feature-specific API calls
│   │   └── index.ts       # Public API
│   └── dashboard/
│       └── ...
├── shared/                # Cross-feature resources
│   ├── components/ui/     # Base UI primitives
│   ├── hooks/             # Shared hooks
│   ├── lib/               # Third-party wrappers
│   └── utils/             # Pure helpers
├── context/               # React Context providers
├── api/                   # Shared API client setup
└── routes.tsx             # Route definitions

Structure rules

  • No cross-feature imports. Feature A cannot import from Feature B's internals. Shared code goes in shared/.
  • Explicit public API. Each feature's index.ts exports only what others need.
  • Flat alternative for small apps. Under ~20 routes, a flat by-concern structure works fine. Migrate to feature-based at 5+ domains.

The skill also generates a CLAUDE.md

When scaffolding a new project, the skill creates a project-specific CLAUDE.md with architecture, commands, and conventions — so future Claude sessions have full context from the start.

Writing Your Own Skills

Use the setup skill as a template. Good skills share these traits:

  • Rich description — include trigger phrases so Claude knows when to activate
  • Opinionated defaults — make decisions so the user doesn't have to
  • Escape hatches — offer alternatives when the default doesn't fit
  • Self-contained — everything Claude needs is in the skill file itself

MCP Servers — External Connections Docs

MCP (Model Context Protocol) servers connect Claude to external tools and data sources — APIs, databases, browsers, task trackers, etc.

Configuration

// ~/.claude.json (global) or .mcp.json (project)
{
  "mcpServers": {
    "my-server": {
      "command": "npx",
      "args": ["-y", "@some/mcp-server"],
      "env": {
        "API_KEY": "your-key"
      }
    }
  }
}

Managing MCP servers

claude mcp add server-name -s user -- command args    # Add
claude mcp list                                        # List
claude mcp remove server-name                          # Remove

Settings & Permissions Docs Permissions

Settings files (most specific wins)

Level File Scope
User ~/.claude/settings.json All projects
Project .claude/settings.json Shared (committed)
Local .claude/settings.local.json Personal (gitignored)

Permission modes

  • default: Ask for most operations
  • plan: Read-only, no edits
  • acceptEdits: Auto-approve file edits, ask for commands
  • dontAsk: Auto-approve everything

Allow rules

{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Read(/home/user/project/**)",
      "Edit(/home/user/project/**)"
    ]
  }
}

Deciding What to Use

When you have a new need, here's how to decide where it goes:

Use CLAUDE.md

"Claude needs to know this about my project"

Architecture, conventions, build commands, schema.

Use Memory

"Claude keeps forgetting my preference"

Save it as a feedback memory so it persists across sessions.

Use a Hook

"After X happens, Claude should do Y"

Post-commit tasks, linting, validation, reminders.

Use a Skill

"When I say X, Claude should do Y"

Deployment, scaffolding, code generation templates. Example: the vite-react-setup-guide skill activates when you discuss new project setup and walks through stack, structure, and conventions.

Use an MCP Server

"Claude needs to talk to an external service"

GitHub, Docker, Git, databases, browsers.

Use a Plugin

"I want to share a bundle of tools"

Package skills + hooks + MCP servers together for your team.

Portable Config with a Private Gist

Your Claude Code config lives in directories that aren't tracked by git. To make it portable across machines, store everything in a private GitHub gist.

Step 1 — Create the gist

# Collect files into a temp dir
mkdir -p /tmp/cc-config
cp .claude/hooks/*.sh /tmp/cc-config/
cp ~/.claude/CLAUDE.md /tmp/cc-config/global-CLAUDE.md

# Create the gist (secret by default)
cd /tmp/cc-config
gh gist create -d "Claude Code personal config" *.sh *.json *.md

Step 2 — Restore on a new machine

# Clone the gist
gh gist clone  /tmp/cc-config

# Restore files
mkdir -p .claude/hooks
cp /tmp/cc-config/*.sh .claude/hooks/
chmod +x .claude/hooks/*.sh

What to store in the gist

File Local path Purpose
Hook scripts .claude/hooks/*.sh Automated workflows
Local settings .claude/settings.local.json Hook registration + permissions
Global CLAUDE.md ~/.claude/CLAUDE.md Memory rules + global instructions

Containers & Isolation Docs

A development container wraps Claude in an isolated environment with its own filesystem, network firewall, and user context. Because the container can't damage your host machine, you can safely run with --dangerously-skip-permissions and let Claude work uninterrupted.

Key idea: Permissions ask "can I do this?" before each action. Containers say "do whatever you want, but only within these walls."

When to use a container

  • Autonomous multi-step tasks — large refactors, feature builds, test generation where approving each command is the bottleneck
  • Untrusted or experimental code — unfamiliar repos, generated code. The firewall prevents data exfiltration
  • CI/CD pipelines — headless mode with no human to approve prompts
  • Team onboarding / demos — consistent pre-configured environment without risking local machines
  • Secure client work — isolate one client's code from your other projects

Containers vs sandboxing

Sandbox (/sandbox)Devcontainer
Isolates Bash tool only — filesystem paths + network domains Entire environment — filesystem, network, processes
Setup One command: /sandbox Docker + devcontainer config
Skip permissions? No — still uses permission system Yes — safe to use --dangerously-skip-permissions
Best for Day-to-day dev with fewer prompts Full autonomy, CI/CD, untrusted repos

Getting started

  1. Install Docker Desktop and the VS Code Dev Containers extension
  2. Copy the .devcontainer/ directory (devcontainer.json, Dockerfile, init-firewall.sh) into your project
  3. Cmd+Shift+P → "Dev Containers: Reopen in Container"
  4. Start Claude in the integrated terminal:
# Full autonomy — safe because you're inside the container
claude --dangerously-skip-permissions

What the container provides

  • Node.js 20 with Claude Code pre-installed
  • ZSH with Powerlevel10k, fzf, git-delta
  • Network firewall — default-deny policy, only allows Anthropic API, GitHub, npm, VS Code marketplace
  • Session persistence — bash history and Claude config survive container restarts
  • Non-root user — runs as node, not root

React/Node customization

For React projects, you may want to add extra domains to the firewall for package CDNs:

# In init-firewall.sh, add to the domain loop:
for domain in \
    "registry.npmjs.org" \
    "api.anthropic.com" \
    "cdn.jsdelivr.net" \
    "unpkg.com" \
    "registry.yarnpkg.com" \
    # ... existing domains ...
; do

Headless mode for CI

# One-shot task in a container
docker run --rm \
  --cap-add=NET_ADMIN --cap-add=NET_RAW \
  -e ANTHROPIC_API_KEY="$ANTHROPIC_API_KEY" \
  -v "$(pwd):/workspace" \
  your-claude-image \
  claude -p "Run npm test and fix any failures" \
  --dangerously-skip-permissions

Warning: Even inside a container, --dangerously-skip-permissions is only safe with trusted repositories. A malicious project could exfiltrate anything accessible inside the container. Always review Claude's changes before merging — git diff is your friend.

Best practices

  1. Commit before launch — clean working tree so you can diff Claude's changes
  2. Scope the task — "Fix the login bug in issue #42" beats "improve the codebase"
  3. Use CLAUDE.md — the container inherits it from your workspace
  4. Pin the version — set CLAUDE_CODE_VERSION in devcontainer.json for CI stability
  5. Minimize the allowlist — every allowed domain is a potential exfiltration path
